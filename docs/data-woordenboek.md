# Data-woordenboek ŌĆö RugFlow ERP

Domeinbegrippen die je moet kennen om dit project te begrijpen.

## Klanten & Commercieel

| Term | Betekenis |
|------|-----------|
| **Debiteur** | Klant/afnemer. Ge├»dentificeerd door `debiteur_nr` (INTEGER). |
| **Afleveradres** | Leveradres van een debiteur. Adres_nr 0 = hoofdadres, 1+ = extra adressen. |
| **Vertegenwoordiger** | Sales representative; **rol op een Medewerker** (`'vertegenwoordiger' = ANY(rollen)`, zie sectie Medewerkers & Rollen). Ge├»dentificeerd via `medewerkers.code` (3-4 letter, UNIQUE). `klanten.vertegenwoordiger_code` en `orders.vertegenwoordiger_code` zijn FKs op deze code. Werkdagen-koppeling als rol-specifieke satelliet (mig 195). V├│├│r ADR-0004 een eigen tabel `vertegenwoordigers`; sindsdien hernoemd naar `medewerkers`. |
| **Betaler** | Debiteur die de factuur betaalt. Kan afwijken van de besteller (self-reference in debiteuren). |
| **Tier** | Klantwaarde-classificatie: Gold (top 10% omzet), Silver (top 30%), Bronze (rest). Berekend via `herbereken_klant_tiers()`. |
| **GLN** | Global Location Number. 13-cijferige code voor bedrijfslocaties (niet voor producten). Staat in debiteuren als `gln_bedrijf`. |
| **GLN afleveradres** | GLN specifiek voor een afleveradres. Gebruikt voor EDI-bestellingen. |
| **GTIN** | Global Trade Item Number. 8/12/13/14-cijferige code voor producten (handelseenheden). EAN-13 valt hieronder. Karpi's GTINs starten met `8715954`. Staat in `producten.ean_code`. Wordt door EDI-partners gebruikt om producten in orderregels te identificeren (`PIA` of `LIN`-segment). |
| **EDI** | Electronic Data Interchange. Gestructureerde digitale berichtuitwisseling tussen bedrijven (orders, orderbevestigingen, facturen, verzendberichten). Bij Karpi via Transus als VAN. |
| **Transus** | EDI-dienstverlener die als full-duplex-vertaler tussen Karpi en handelspartners zit. Karpi spreekt fixed-width "Custom ERP", partners spreken EDIFACT ŌĆö Transus vertaalt beide kanten op. Productie-endpoint: `https://webconnect.transus.com/exchange.asmx`. |
| **EDIFACT** | UN/EDIFACT ŌĆö internationale EDI-standaard van de VN. Berichten als `ORDERS`, `ORDRSP`, `INVOIC`, `DESADV`. Karpi's partners (BDSK, Ostermann, XXXLUTZ, Hornbach) gebruiken D96A EANCOM (subset voor retail). Karpi zelf "spreekt" alleen het door Transus vertaalde fixed-width. |
| **Custom ERP (Transus)** | Naam van Karpi's gegevensbron-config in Transus Online (ID 17653, versie 10, type "Fixed length"). Definieert welke velden uit een EDIFACT-bericht naar welke kolompositie in onze `.inh`-output gaan. |
| **TransusXML** | XML-bronformaat dat Transus accepteert voor uitgaande orderbevestigingen (`<ORDERRESPONSES>`). RugFlow bouwt dit voor `orderbev` en Transus vertaalt het naar EDIFACT `ORDRSP` voor de handelspartner. |
| **Karpi fixed-width INVOIC** | Uitgaand bronformaat voor EDI-facturen. RugFlow bouwt een 1107-byte headerregel + 312-byte artikelregels; Transus vertaalt dit naar EDIFACT D96A `INVOIC` (`BGM+380`). Wordt gequeued als `edi_berichten.berichttype='factuur'`. |
| **M10100 / M10110 / M10300** | De drie SOAP-methodes van de Transus API. M10100 = bericht versturen, M10110 = bericht ontvangen (poll-based, geen webhook), M10300 = ontvangst bevestigen (anders krijg je hetzelfde bericht eindeloos terug). |
| **TransactionID** | Uniek ID per Transus-bericht (zowel inkomend als uitgaand). Bron-van-waarheid voor idempotentie in `edi_berichten.transactie_id`. |
| **MessageReference** | Unieke ID per logisch EDIFACT-bericht (UNH-segment). Een Transus-interchange (UNB...UNZ) kan meerdere UNH's bevatten; Transus splitst die in losse berichten richting Karpi (├®├®n TransactionID per UNH). |
| **IsTestMessage** | Vlag op berichtniveau (`Y`/`N`) die aangeeft dat een bericht voor test-doeleinden is. Karpi mag dan geen echte order aanmaken. Per-partner activeerbaar via `edi_handelspartner_config.test_modus`. |
| **NAD-segmenten** | EDIFACT-partij-aanduidingen. `NAD+SU` = Supplier (Karpi), `NAD+BY` = Buyer (winkel/besteller), `NAD+IV` = Invoicee (gefactureerde HQ), `NAD+DP` = Delivery Party (afleveradres), `NAD+SN` = Ship Notify, `NAD+ZZZ` = mutually-defined. Een EDI-order kan tot 4 verschillende GLN's hebben ŌĆö gemodelleerd op `orders` als `besteller_gln`/`factuuradres_gln`/`afleveradres_gln` + `bes_*`-snapshots. |
| **BGM-codes** | EDIFACT bericht-doel: 220 = Order, 231 = Orderbevestiging, 351 = Despatch Advice (verzendbericht), 380 = Commercial Invoice. |
| **edi_berichten** | Centrale audit-/queue-tabel. Inkomend (`richting='in'`) door `transus-poll` edge function. Uitgaand (`richting='uit'`) door triggers op orders/facturen/zendingen + `transus-send` edge function. Status: `Wachtrij` ŌåÆ `Bezig` ŌåÆ `Verstuurd` / `Verwerkt` / `Fout` / `Geannuleerd`. |
| **edi_handelspartner_config** | Per debiteur welke EDI-berichttypen actief zijn (`order_in`, `orderbev_uit`, `factuur_uit`, `verzend_uit`) + de hoofdschakelaar `transus_actief` en de cutover-modus `test_modus`. Komt overeen met de toggles per partner in Transus Online ŌåÆ Handelspartners ŌåÆ Processen. **N.B.** Tot ADR-0008 stond hier ook een `vervoerder_code`-kolom (mig 170) die de klant-fallback voor vervoerder-keuze huisvestte; die kolom is gedropt ŌĆö vervoerder-keuze leeft nu volledig in `vervoerder_selectie_regels` + `order_regels.vervoerder_code`. |
| **Prijslijst** | Per klant een prijslijst (header + regels). Bepaalt welke prijs een klant betaalt per artikel. |
| **Betaalconditie** | Betalingstermijn, bijv. "30 dagen netto". |
| **Inkoopgroep** (vroeger "Inkooporganisatie") | Centrale inkooporganisatie (INKC-code) waaronder een groep debiteuren samen inkoopt en daardoor aan een gedeelde prijslijst of korting hangt. 10 bekende groepen (BEGROS, DECOR UNION, FACHHANDELSRING, INTERRING, VME, VME (TH), TINTTO, INHOUSE, HOUSE OF DUTCHZ, MUSTERRING). Eigen tabel + FK `debiteuren.inkoopgroep_code` sinds mig 189. Op `orders` blijft `inkooporganisatie` als TEXT-snapshot ŌĆö orders bewegen niet mee als de groep van een klant wijzigt. |
| **Standaard levertermijn** | Twee aparte getallen: `standaard_maat_werkdagen` (default 5, kalenderdagen, voor karpetten uit voorraad) en `maatwerk_weken` (default 4, voor gesneden+geconfectioneerde karpetten). Globale defaults in `app_config.order_config`; per klant overschrijfbaar. Order-afleverdatum wordt automatisch berekend als max van beide typen in de order. |
| **Verzendweek** | De ISO-week waarin een order naar buiten gaat. Karpi communiceert leverbeloftes als week (niet als specifieke dag), dus de UI toont uitsluitend de week. Onderliggend blijft `orders.afleverdatum` een DATE (vrijdag van de gekozen week) zodat alle bestaande logica (mig 153 IO-claim sync, pick & ship bucket, levertijd-berekening, sortering) ongewijzigd kan blijven. Centrale helpers in [`lib/orders/verzendweek.ts`](../frontend/src/lib/orders/verzendweek.ts): `verzendWeekVoor`, `verzendWeekIsoString`, `verzendWeekStringToDatum`. **Pick-window**: een order met verzendweek N moet in week N-1 in de pick-lijst verschijnen ŌĆö dat regelt `bucketVoor` in [`magazijn/lib/buckets.ts`](../frontend/src/modules/magazijn/lib/buckets.ts) (afleverdatum < maandag-over-volgende-week ŌåÆ `'deze_week'`-bucket). |
| **Deelleveringen** | Per klant boolean (`debiteuren.deelleveringen_toegestaan`, default FALSE). Als TRUE en een order bevat zowel standaard-maat als maatwerk regels: bij aanmaken wordt de order opgesplitst in 2 losse orders ŌĆö ├®├®n met de standaard-regels (korte levertermijn) en ├®├®n met de maatwerk-regels (lange levertermijn). Verzendkosten-regel gaat mee met de standaard-order. |
| **Verzameldebiteur** | Debiteuren-rij die een groep externe eindklanten vertegenwoordigt. Voor webshop-orders gebruikt RugFlow de bestaande **debiteur 260000 "FLOORPASSION"** ŌĆö particuliere kopers krijgen geen eigen debiteuren-rij, hun naam/adres landt in de `afl_*`-snapshotvelden op de order. (Synthetische rij 99001 uit migratie 091 is niet in productie: keuze viel op 260000 om aan te sluiten bij het oude systeem.) |
| **Webshop-order** | Order die automatisch is aangemaakt door de Lightspeed webhook-integratie (of via batch-import uit het oude systeem). Herkenbaar aan `orders.bron_systeem = 'lightspeed'`. Valt onder debiteur 260000; de shop van herkomst staat in `bron_shop` (`floorpassion_nl` / `floorpassion_de`). |
| **Bron-systeem** | Veld `orders.bron_systeem` dat aangeeft waar een order vandaan komt. NULL = handmatig in RugFlow aangemaakt. Bekende externe bronnen: `'lightspeed'` (Floorpassion webshops), `'edi'` (Transus-koppeling ŌĆö fase 2 vult dit). Toekomstig mogelijk: `'marketplace'`. Samen met `bron_order_id` uniek voor idempotentie bij webhook/poll-retries. |
| **Unmatched-regel** | Orderregel waar geen `artikelnr` aan kon worden gekoppeld. Herkenbaar aan een prefix in `omschrijving`: `[UNMATCHED]` (onbekend), `[STAAL]` (gratis muster/sample), `[MAATWERK]` (Wunschgr├Č├¤e / op maat / volgens tekening), `[MAATWERK-ROND]` (ronde maatwerk met diameter). Order krijgt `heeft_unmatched_regels = TRUE`, wat in de orderlijst als actie-vereist-badge zichtbaar moet worden. |
| **`heeft_unmatched_regels`** | Boolean-vlag op `orders`. TRUE zodra minstens 1 regel `artikelnr IS NULL` heeft. Automatisch onderhouden door trigger op `order_regels` (migratie 094) + expliciete set in `create_webshop_order` RPC. Index `orders_heeft_unmatched_idx` voor snel filteren van review-orders. |

## Producten & Voorraad

| Term | Betekenis |
|------|-----------|
| **Artikelnr** | Unieke productcode (TEXT). PK in producten-tabel. |
| **Karpi-code** | Samengestelde code: kwaliteit + kleur + afmetingen. Bijv. `CISC21XX160230`. |
| **Kwaliteit** | Tapijtsoort, gecodeerd in 3-4 letters. Bijv. CISC = Cisco, BEAC = Beach Life, MIRA = Mirage. |
| **Kwaliteitscode** | De 3-4 letter code. Eerste letters uit de karpi_code. |
| **Kleur-code** | Eerste 2 cijfers uit de karpi_code na de kwaliteitscode. |
| **Zoeksleutel** | kwaliteit_code + "_" + kleur_code. Bijv. "CISC_21". Gebruikt voor gecombineerd zoeken. |
| **Collectie** | Canonieke groep van **aliassen voor ├®├®n fysiek tapijt-type**. Bijv. "Vernissage/Lago" bevat VERI, LAGO, GLOR ŌĆö fysiek dezelfde partij, andere namen voor verschillende afnemers/markten. 56 groepen, 170 codes. **Bron-van-waarheid voor uitwisselbaarheid; bewerkt via Producten ŌåÆ tab Uitwisselbaar.** |
| **Uitwisselbaar** | Twee `(kwaliteit_code, kleur_code)`-paren zijn uitwisselbaar wanneer (a) beide kwaliteiten dezelfde `collectie_id` hebben **├®n** (b) de kleur-codes na normalisatie (`normaliseer_kleur_code()`, strip trailing `.0`) gelijk zijn. Resolver: SQL-functie `uitwisselbare_paren(kw, kl)` ŌĆö alle voorraad-/order-/snijplan-callers gaan hierdoor. Een paar zonder partners ("niet-overeenkomend" in UI) is uitwisselbaar met zichzelf. |
| **Aliassing-lagen** | Drie niveaus van naam-versus-fysieke-identiteit: (1) **collectie** = markt/leverancier-niveau aliassing, ├®├®n partij onder N namen; (2) **`klanteigen_namen`** = klant-niveau display-naam, per debiteur; (3) **stickering** = bij rol-output krijgt het stuk de naam uit `order_regel.maatwerk_kwaliteit_code`. Voor rollen is uitwisseling kosteloos (sticker komt pas na snijden); voor vaste-maat producten kost het een sticker-wissel ("liever sticker veranderen dan nee-verkopen"). |
| **Rol** | Individuele fysieke tapijtrol in het magazijn. Elke rol heeft een uniek `rolnummer`, specifieke afmetingen en waarde. |
| **Rolnummer** | Unieke identifier per fysieke rol. Legacy: puur numeriek (bv. `109801`), S-prefix (bv. `S0375-1CBON`) of andere formaten uit de oude administratie. **Nieuw (migratie 135)**: rollen die via `boek_ontvangst` aangemaakt worden krijgen automatisch `R-YYYY-NNNN` (bv. `R-2026-0001`) uit sequence `r_2026_seq` via `volgend_nummer('R')` ŌĆö consistent met `ORD-`/`INK-`/`SNIJ-`-nummering. |
| **VVP** | Verkoopprijs per vierkante meter (Verkoop Vaste Prijs per m2). |
| **Vrije voorraad** | Voorraad minus gereserveerd minus backorder + besteld inkoop. Wat daadwerkelijk beschikbaar is. |
| **Volle rol** | Rol met standaard breedte ├®n volledige lengte. `rol_type = 'volle_rol'`. Standaard breedte komt primair uit `kwaliteiten.standaard_breedte_cm` (bron van waarheid sinds migratie 086), fallback op laatste 3 cijfers artikelnr, daarna 400 cm. |
| **Aangebroken rol** | Rol met standaard breedte maar reeds aangesneden (kortere lengte, Ōēź100 cm). Ontstaat na `voltooi_snijplan_rol()`. `rol_type = 'aangebroken'`, `status = 'beschikbaar'`. |
| **Reststuk** | Rol met afwijkende (smallere) breedte t.o.v. standaard, ├│f met lengte <100 cm. `rol_type = 'reststuk'`. Classificatie gebeurt automatisch via trigger o.b.v. `bereken_rol_type()`. |
| **Standaard rolbreedte** | Per kwaliteit vastgelegd in `kwaliteiten.standaard_breedte_cm`. Voorbeelden: 400 cm (default, CISC/GALA/VERR/ŌĆ”), 320 cm (OASI/NOMA/RUBI/CAVA/EMIR/ŌĆ”), 200 cm (DUBE/VETB/CLAS), 147-180 cm (BUX-serie lopers), 160 cm (BEAC/BEAB), 500 cm (CROW). |
| **Artikelnr-codering** | Veel (niet alle) Karpi-artikelnummers volgen: 4 letters (kwaliteit) + 2 cijfers (kleur) + 3 cijfers (breedte in cm). Voorbeeld: `CISC12400` = CISC kwaliteit, kleur 12, breedte 400 cm. Voor kwaliteiten zonder dit suffix (OASI-serie e.d.) geldt uitsluitend `kwaliteiten.standaard_breedte_cm`. |
| **Voorraadpositie per (Kwaliteit, Kleur)** | E├®n domeinconcept dat per `(kwaliteit_code, kleur_code)`-paar antwoord geeft op drie vragen: (1) **eigen voorraad** ŌĆö wat heb ik vandaag uit eigen rol (volle/aangebroken/reststuk + totaal m┬▓); (2) **uitwisselbare partners** ŌĆö welke andere paren in dezelfde collectie kunnen dit aanvullen, plus de "beste partner" (top-m┬▓-partner, alleen ingevuld wanneer eigen voorraad = 0); (3) **besteld** ŌĆö wat komt binnenkort uit inkoop (m, m┬▓, orders_count, eerstvolgende leverweek/datum). Bron: SQL-RPC `voorraadposities(p_kwaliteit, p_kleur, p_search)` (mig 179) en frontend-Module [`@/modules/voorraadpositie`](../frontend/src/modules/voorraadpositie/) die als enige seam tussen UI en deze drie sub-bronnen staat. Vervangt op termijn de losse callers van `rollen_uitwissel_voorraad`, `uitwisselbare_partners` en `besteld_per_kwaliteit_kleur`. Kleur-normalisatie (`'15.0' ŌåÆ '15'`) is bron-van-waarheid in zowel SQL (regexp_replace) als TS (`normaliseerKleurcode`). |
| **Gewicht/m┬▓** | Density per kwaliteit in kg per vierkante meter. Bron-van-waarheid: `kwaliteiten.gewicht_per_m2_kg` (mig 184). Drijft alle gewicht-berekeningen voor orderregels en zendingen via de gewicht-resolver-Module. Ingegeven via instellingen-pagina `/instellingen/kwaliteiten` of via eenmalige Excel-import (`import/import_kwaliteit_gewichten.py`). NULL = nog niet ingevuld; producten in deze kwaliteit vallen terug op legacy `producten.gewicht_kg` met flag `gewicht_uit_kwaliteit=false`. |
| **Gewicht-resolver** | SQL-Module die gewicht voor een orderregel of product berekent uit oppervlakte ├Ś kwaliteit-density. **Smal publiek interface (drie functies, mig 185):** `gewicht_per_m2_voor_kwaliteit(code)`, `bereken_product_gewicht_kg(artikelnr)`, `bereken_orderregel_gewicht_kg(id)`. **Brede implementatie (intern):** oppervlak-bepaling per producttype, NULL-fallback, trigger-cascade kwaliteit ŌåÆ producten ŌåÆ open order_regels. Alle gewicht-callers gaan hierdoor ŌĆö geen verspreide `oppervlak ├Ś density`-formules meer. |
| **Gewicht-cache** | Verzamelnaam voor de gederiveerde gewicht-velden: `producten.gewicht_kg` (kg per stuk, gederiveerd uit kwaliteit-density ├Ś `lengte_cm ├Ś breedte_cm`) en `order_regels.gewicht_kg` (snapshot bij regel-aanmaak/-update, herrekend voor open orders bij kwaliteit-mutatie via `trg_kwaliteit_gewicht_recalc` + `trg_product_gewicht_recalc`, mig 185). Beide onderhouden door triggers ŌĆö niet handmatig editen. Verzonden orders bevroren via `zendingen.totaal_gewicht_kg` (mig 176/177). |
| **Gewicht-uit-kwaliteit-flag** | Boolean `producten.gewicht_uit_kwaliteit` (mig 184). TRUE = `gewicht_kg` gederiveerd uit `kwaliteiten.gewicht_per_m2_kg`. FALSE = legacy waarde uit oude systeem (kwaliteit heeft nog geen gewicht ingevuld, of artikelnr-patroon was niet parsbaar). Maakt migratie-voortgang zichtbaar; data-completing-rapport in `/instellingen/kwaliteiten` filtert hierop. UI toont badge `<GewichtBronBadge>` op product-detail bij FALSE. |
| **Bbox-oppervlak (gewicht)** | Voor maatwerk-vormen (rond/ovaal) wordt het gewicht berekend op het omsluitende rechthoek (lengte ├Ś breedte), niet op de werkelijke geometrische oppervlakte. Industrie-standaard voor zowel prijs ([`berekenPrijsOppervlakM2`](../frontend/src/lib/utils/maatwerk-prijs.ts)) als gewicht. Voor rond: `diameter┬▓`. Pragmatische match met klantverwachting + materiaal-verbruik: een rond kleed wordt uit een vierkant gesneden, materiaal en gewicht zijn gelijk aan dat vierkant. **Let op**: voor catalogus-producten met `vorm='rond'` (mig 188) gebruikt `bereken_product_gewicht_kg` juist het echte cirkel-oppervlak `ŽĆ ├Ś (d/200)┬▓`, niet de bbox ŌĆö andere keuze dan maatwerk omdat catalogus-gewicht gemeten wordt op de fabriek (geen snij-marge). Voor catalogus-OVL blijft bbox de aanname. |
| **Vorm (producten)** | `producten.vorm` Ōłł `{rechthoek, rond}` (mig 188). Bepaalt welke formule de gewicht-resolver gebruikt voor catalogus-artikelen. RND-codes uit `karpi_code` (suffix `\d{3}RND`) krijgen `vorm='rond'`; alle andere artikelen ŌĆö incl. ovale ŌĆö blijven `rechthoek` met bbox-formule. Niet te verwarren met de **vorm-keuze** in maatwerk-orderregels (mig 179-183), die werkt op `order_regels.vorm` met meer waardes voor klantkeuze (rond/ovaal/halfrond/etc.) en Ōé¼75 toeslag. |
| **Bandkleur (label)** | Master-string voor de afwerking-band die op een maatwerk-tapijt komt ŌĆö bv. "Piero Taupe 431". E├®n tabel `afwerking_kleuren` per afwerking met `(afwerking_code, label)` als unique key (mig 194). Vervangt de pre-mig-194 hardcoded `Piero `-prefix in de UI en de losse `band_merk`/`band_omschrijving`/`band_kleur`-velden in `maatwerk_band_defaults`. **Default per kwaliteit/kleur** zit in `maatwerk_band_defaults.afwerking_kleur_id`, beheerd via /producten kleur-uitvouw. **Order-form** gebruikt strict-FK-dropdown ŌĆö geen vrije tekst meer. **Bestaande order-regels** behouden hun tekst-snapshot in `order_regels.maatwerk_band_kleur`; nieuwe regels schrijven de FK in `maatwerk_band_kleur_id` ├®n tegelijk de label-string in de tekstkolom. |

| **Maatwerk-prijs-formule** | `oppervlak_m┬▓ ├Ś m┬▓-prijs + vorm-toeslag + afwerkingprijs ŌłÆ korting%`. Pure TS in `modules/maatwerk/lib/prijs.ts` (was `lib/utils/maatwerk-prijs.ts`). | **Bbox-oppervlak (maatwerk)** | Oppervlak voor maatwerk-prijs en -gewicht: rechthoek = l ├Ś b, ovaal = l ├Ś b (omsluitende rechthoek), rond = d┬▓ (omsluitend vierkant).
