# Data-woordenboek — RugFlow ERP

Domeinbegrippen die je moet kennen om dit project te begrijpen.

## Klanten & Commercieel

| Term | Betekenis |
|------|-----------|
| **Debiteur** | Klant/afnemer. Geïdentificeerd door `debiteur_nr` (INTEGER). |
| **Debiteur-Module** | De Module die masterdata, afleveradressen, klanteigen namen en klant-artikelnummers bezit. Frontend-folder: `modules/debiteuren/` (DB-aligned naam — UI-tekst blijft "Klant", routes blijven `/klanten/...`). Bezit het concept Debiteur als eigenaar van klant-bound data. Cross-cuts: tier-berekening blijft SQL-cron, vertegenwoordiger-koppeling consumeert Medewerkers-Module, adres-snapshot blijft order-creatie-territorium. **Slot-component `<KlantBenaming/>`** is de smalle interface voor cross-Module display van klant-bound benaming-resolutie (orders, facturatie, magazijn) — backend-callers (factuur-RPC, EDI-builder, pakbon-edge) consumeren `resolve_klanteigen_naam`-RPC direct. Slot-tabs Orders/Prijslijst tussentijds via directe imports tot ADR-0001-uitvoering / Prijslijst-ADR. Zie [ADR-0011](adr/0011-debiteur-als-deep-module.md). |
| **Afleveradres** | Leveradres van een debiteur. Adres_nr 0 = hoofdadres, 1+ = extra adressen. |
| **Vertegenwoordiger** | Sales representative; **rol op een Medewerker** (`'vertegenwoordiger' = ANY(rollen)`, zie sectie Medewerkers & Rollen). Geïdentificeerd via `medewerkers.code` (3-4 letter, UNIQUE). `klanten.vertegenwoordiger_code` en `orders.vertegenwoordiger_code` zijn FKs op deze code. Werkdagen-koppeling als rol-specifieke satelliet (mig 195). Vóór ADR-0004 een eigen tabel `vertegenwoordigers`; sindsdien hernoemd naar `medewerkers`. |
| **Betaler** | Debiteur die de factuur betaalt. Kan afwijken van de besteller (self-reference in debiteuren). |
| **Tier** | Klantwaarde-classificatie: Gold (top 10% omzet), Silver (top 30%), Bronze (rest). Berekend via `herbereken_klant_tiers()`. |
| **GLN** | Global Location Number. 13-cijferige code voor bedrijfslocaties (niet voor producten). Staat in debiteuren als `gln_bedrijf`. |
| **GLN afleveradres** | GLN specifiek voor een afleveradres. Gebruikt voor EDI-bestellingen. |
| **GTIN** | Global Trade Item Number. 8/12/13/14-cijferige code voor producten (handelseenheden). EAN-13 valt hieronder. Karpi's GTINs starten met `8715954`. Staat in `producten.ean_code`. Wordt door EDI-partners gebruikt om producten in orderregels te identificeren (`PIA` of `LIN`-segment). |
| **EDI** | Electronic Data Interchange. Gestructureerde digitale berichtuitwisseling tussen bedrijven (orders, orderbevestigingen, facturen, verzendberichten). Bij Karpi via Transus als VAN. |
| **Transus** | EDI-dienstverlener die als full-duplex-vertaler tussen Karpi en handelspartners zit. Karpi spreekt fixed-width "Custom ERP", partners spreken EDIFACT — Transus vertaalt beide kanten op. Productie-endpoint: `https://webconnect.transus.com/exchange.asmx`. |
| **EDIFACT** | UN/EDIFACT — internationale EDI-standaard van de VN. Berichten als `ORDERS`, `ORDRSP`, `INVOIC`, `DESADV`. Karpi's partners (BDSK, Ostermann, XXXLUTZ, Hornbach) gebruiken D96A EANCOM (subset voor retail). Karpi zelf "spreekt" alleen het door Transus vertaalde fixed-width. |
| **Custom ERP (Transus)** | Naam van Karpi's gegevensbron-config in Transus Online (ID 17653, versie 10, type "Fixed length"). Definieert welke velden uit een EDIFACT-bericht naar welke kolompositie in onze `.inh`-output gaan. |
| **TransusXML** | XML-bronformaat dat Transus accepteert voor uitgaande orderbevestigingen (`<ORDERRESPONSES>`). RugFlow bouwt dit voor `orderbev` en Transus vertaalt het naar EDIFACT `ORDRSP` voor de handelspartner. |
| **Karpi fixed-width INVOIC** | Uitgaand bronformaat voor EDI-facturen. RugFlow bouwt een 1107-byte headerregel + 312-byte artikelregels; Transus vertaalt dit naar EDIFACT D96A `INVOIC` (`BGM+380`). Wordt gequeued als `edi_berichten.berichttype='factuur'`. |
| **M10100 / M10110 / M10300** | De drie SOAP-methodes van de Transus API. M10100 = bericht versturen, M10110 = bericht ontvangen (poll-based, geen webhook), M10300 = ontvangst bevestigen (anders krijg je hetzelfde bericht eindeloos terug). |
| **TransactionID** | Uniek ID per Transus-bericht (zowel inkomend als uitgaand). Bron-van-waarheid voor idempotentie in `edi_berichten.transactie_id`. |
| **MessageReference** | Unieke ID per logisch EDIFACT-bericht (UNH-segment). Een Transus-interchange (UNB...UNZ) kan meerdere UNH's bevatten; Transus splitst die in losse berichten richting Karpi (één TransactionID per UNH). |
| **IsTestMessage** | Vlag op berichtniveau (`Y`/`N`) die aangeeft dat een bericht voor test-doeleinden is. Karpi mag dan geen echte order aanmaken. Per-partner activeerbaar via `edi_handelspartner_config.test_modus`. |
| **NAD-segmenten** | EDIFACT-partij-aanduidingen. `NAD+SU` = Supplier (Karpi), `NAD+BY` = Buyer (winkel/besteller), `NAD+IV` = Invoicee (gefactureerde HQ), `NAD+DP` = Delivery Party (afleveradres), `NAD+SN` = Ship Notify, `NAD+ZZZ` = mutually-defined. Een EDI-order kan tot 4 verschillende GLN's hebben — gemodelleerd op `orders` als `besteller_gln`/`factuuradres_gln`/`afleveradres_gln` + `bes_*`-snapshots. |
| **BGM-codes** | EDIFACT bericht-doel: 220 = Order, 231 = Orderbevestiging, 351 = Despatch Advice (verzendbericht), 380 = Commercial Invoice. |
| **edi_berichten** | Centrale audit-/queue-tabel. Inkomend (`richting='in'`) door `transus-poll` edge function. Uitgaand (`richting='uit'`) door triggers op orders/facturen/zendingen + `transus-send` edge function. Status: `Wachtrij` → `Bezig` → `Verstuurd` / `Verwerkt` / `Fout` / `Geannuleerd`. |
| **edi_handelspartner_config** | Per debiteur welke EDI-berichttypen actief zijn (`order_in`, `orderbev_uit`, `factuur_uit`, `verzend_uit`) + de hoofdschakelaar `transus_actief` en de cutover-modus `test_modus`. Komt overeen met de toggles per partner in Transus Online → Handelspartners → Processen. **N.B.** Tot ADR-0008 stond hier ook een `vervoerder_code`-kolom (mig 170) die de klant-fallback voor vervoerder-keuze huisvestte; die kolom is gedropt — vervoerder-keuze leeft nu volledig in `vervoerder_selectie_regels` + `order_regels.vervoerder_code`. |
| **Prijslijst** | Per klant een prijslijst (header + regels). Bepaalt welke prijs een klant betaalt per artikel. |
| **Betaalconditie** | Betalingstermijn, bijv. "30 dagen netto". |
| **Inkoopgroep** (vroeger "Inkooporganisatie") | Centrale inkooporganisatie (INKC-code) waaronder een groep debiteuren samen inkoopt en daardoor aan een gedeelde prijslijst of korting hangt. 10 bekende groepen (BEGROS, DECOR UNION, FACHHANDELSRING, INTERRING, VME, VME (TH), TINTTO, INHOUSE, HOUSE OF DUTCHZ, MUSTERRING). Eigen tabel + FK `debiteuren.inkoopgroep_code` sinds mig 189. Op `orders` blijft `inkooporganisatie` als TEXT-snapshot — orders bewegen niet mee als de groep van een klant wijzigt. |
| **Standaard levertermijn** | Twee aparte getallen: `standaard_maat_werkdagen` (default 5, kalenderdagen, voor karpetten uit voorraad) en `maatwerk_weken` (default 4, voor gesneden+geconfectioneerde karpetten). Globale defaults in `app_config.order_config`; per klant overschrijfbaar. Order-afleverdatum wordt automatisch berekend als max van beide typen in de order. |
| **Verzendweek** | De ISO-week waarin een order naar buiten gaat. Karpi communiceert leverbeloftes als week (niet als specifieke dag), dus de UI toont uitsluitend de week. Onderliggend blijft `orders.afleverdatum` een DATE (vrijdag van de gekozen week) zodat alle bestaande logica (mig 153 IO-claim sync, pick & ship bucket, levertijd-berekening, sortering) ongewijzigd kan blijven. De **ISO-week-rekenkern** woont sinds 2026-06-07 in [`lib/utils/iso-week.ts`](../frontend/src/lib/utils/iso-week.ts) (UTC-correct, TZ-onafhankelijk; spiegelt SQL `to_char IW` mig 145/228 en de Deno-kern [`_shared/iso-week.ts`](../supabase/functions/_shared/iso-week.ts)). De domein-/label-helpers in [`lib/orders/verzendweek.ts`](../frontend/src/lib/orders/verzendweek.ts) (`verzendWeekVoor`, `verzendWeekIsoString`, `verzendWeekStringToDatum`, NL-labels, pick-week-regels) **consumeren** die kern — niet meer hun eigen weekberekening. **Pick-window**: een order met verzendweek N moet in week N-1 in de pick-lijst verschijnen — dat regelt `bucketVoor` in [`magazijn/lib/buckets.ts`](../frontend/src/modules/magazijn/lib/buckets.ts) (afleverdatum < maandag-over-volgende-week → `'deze_week'`-bucket). |
| **Deelleveringen** | Per klant boolean (`debiteuren.deelleveringen_toegestaan`, default FALSE). Als TRUE en een order bevat zowel standaard-maat als maatwerk regels: bij aanmaken wordt de order opgesplitst in 2 losse orders — één met de standaard-regels (korte levertermijn) en één met de maatwerk-regels (lange levertermijn). Verzendkosten-regel gaat mee met de standaard-order. |
| **Verzameldebiteur** | Debiteuren-rij die een groep externe eindklanten vertegenwoordigt. Voor webshop-orders gebruikt RugFlow de bestaande **debiteur 260000 "FLOORPASSION"** — particuliere kopers krijgen geen eigen debiteuren-rij, hun naam/adres landt in de `afl_*`-snapshotvelden op de order. (Synthetische rij 99001 uit migratie 091 is niet in productie: keuze viel op 260000 om aan te sluiten bij het oude systeem.) |
| **Webshop-order** | Order die automatisch is aangemaakt door de Lightspeed webhook-integratie (of via batch-import uit het oude systeem). Herkenbaar aan `orders.bron_systeem = 'lightspeed'`. Valt onder debiteur 260000; de shop van herkomst staat in `bron_shop` (`floorpassion_nl` / `floorpassion_de`). |
| **Bron-systeem** | Veld `orders.bron_systeem` dat aangeeft waar een order vandaan komt. NULL = handmatig in RugFlow aangemaakt. Bekende externe bronnen: `'lightspeed'` (Floorpassion webshops), `'edi'` (Transus-koppeling — fase 2 vult dit). Toekomstig mogelijk: `'marketplace'`. Samen met `bron_order_id` uniek voor idempotentie bij webhook/poll-retries. |
| **Unmatched-regel** | Orderregel waar geen `artikelnr` aan kon worden gekoppeld. Herkenbaar aan een prefix in `omschrijving`: `[UNMATCHED]` (onbekend), `[STAAL]` (gratis muster/sample), `[MAATWERK]` (Wunschgröße / op maat / volgens tekening), `[MAATWERK-ROND]` (ronde maatwerk met diameter). Order krijgt `heeft_unmatched_regels = TRUE`, wat in de orderlijst als actie-vereist-badge zichtbaar moet worden. |
| **`heeft_unmatched_regels`** | Boolean-vlag op `orders`. TRUE zodra minstens 1 regel `artikelnr IS NULL` heeft. Automatisch onderhouden door trigger op `order_regels` (migratie 094) + expliciete set in `create_webshop_order` RPC. Index `orders_heeft_unmatched_idx` voor snel filteren van review-orders. |

## Producten & Voorraad

| Term | Betekenis |
|------|-----------|
| **Artikelnr** | Unieke productcode (TEXT). PK in producten-tabel. |
| **Karpi-code** | Samengestelde code: kwaliteit + kleur + afmetingen. Bijv. `CISC21XX160230`. |
| **Kwaliteit** | Tapijtsoort, gecodeerd in 3-4 letters. Bijv. CISC = Cisco, BEAC = Beach Life, MIRA = Mirage. |
| **Kwaliteitscode** | De 3-4 letter code. Eerste letters uit de karpi_code. |
| **Kleur-code** | Eerste 2 cijfers uit de karpi_code na de kwaliteitscode. |
| **Zoeksleutel** | kwaliteit_code + "_" + kleur_code. Bijv. "CISC_21". Gebruikt voor gecombineerd zoeken. |
| **Collectie** | Canonieke groep van **aliassen voor één fysiek tapijt-type**. Bijv. "Vernissage/Lago" bevat VERI, LAGO, GLOR — fysiek dezelfde partij, andere namen voor verschillende afnemers/markten. 56 groepen, 170 codes. **Bron-van-waarheid voor uitwisselbaarheid; bewerkt via Producten → tab Uitwisselbaar.** |
| **Uitwisselbaar** | Twee `(kwaliteit_code, kleur_code)`-paren zijn uitwisselbaar wanneer (a) beide kwaliteiten dezelfde `collectie_id` hebben **én** (b) de kleur-codes na normalisatie (`normaliseer_kleur_code()`, strip trailing `.0`) gelijk zijn. Resolver: SQL-functie `uitwisselbare_paren(kw, kl)` — alle voorraad-/order-/snijplan-callers gaan hierdoor. Een paar zonder partners ("niet-overeenkomend" in UI) is uitwisselbaar met zichzelf. |
| **Aliassing-lagen** | Drie niveaus van naam-versus-fysieke-identiteit: (1) **collectie** = markt/leverancier-niveau aliassing, één partij onder N namen; (2) **`klanteigen_namen`** = klant-niveau display-naam, per debiteur; (3) **stickering** = bij rol-output krijgt het stuk de naam uit `order_regel.maatwerk_kwaliteit_code`. Voor rollen is uitwisseling kosteloos (sticker komt pas na snijden); voor vaste-maat producten kost het een sticker-wissel ("liever sticker veranderen dan nee-verkopen"). |
| **Rol** | Individuele fysieke tapijtrol in het magazijn. Elke rol heeft een uniek `rolnummer`, specifieke afmetingen en waarde. |
| **Rolnummer** | Unieke identifier per fysieke rol. Legacy: puur numeriek (bv. `109801`), S-prefix (bv. `S0375-1CBON`) of andere formaten uit de oude administratie. **Nieuw (migratie 135)**: rollen die via `boek_ontvangst` aangemaakt worden krijgen automatisch `R-YYYY-NNNN` (bv. `R-2026-0001`) uit sequence `r_2026_seq` via `volgend_nummer('R')` — consistent met `ORD-`/`INK-`/`SNIJ-`-nummering. |
| **VVP** | Verkoopprijs per vierkante meter (Verkoop Vaste Prijs per m2). |
| **Vrije voorraad** | Voorraad minus gereserveerd minus backorder + besteld inkoop. Wat daadwerkelijk beschikbaar is. |
| **Bruto-maatwerkvraag** | Per uitwisselbare familie de **pessimistische projectie** van rol-m² die nog uit voorraad gesneden moet worden voor open maatwerk-snijplannen. Bewust **per-stuk** berekend (geen packer-savings, geen snij-marges — die zitten al in het ingebouwde pessimisme van de formule), geen tijdshorizon-filter ("staat los van de snijplanning die 4 weken vooruitkijkt"). **Formule per stuk:** `min(stuk.lengte_cm, stuk.breedte_cm) × kwaliteit.standaard_breedte_cm`, gesommeerd over snijplannen in `{Wacht, Gepland, Snijden}`, geaggregeerd op familie-sleutel `(collectie_id, genormaliseerde_kleur_code)` — dezelfde resolver als [[Uitwisselbaar]]. **Niet te verwarren met** [[Claim]] / `producten.gereserveerd`: dat is **fysiek toegewezen materiaal**, dit is een **planning-projectie**. Geen rij in `order_reserveringen`; berekend in een RPC (uitbreiding van `voorraadposities` of zuster-RPC), niet gepersisteerd. Bron voor [[Vrij voor nieuw maatwerk]] op Rollen & Reststukken-pagina. |
| **Vrij voor nieuw maatwerk** | Afgeleide KPI per uitwisselbare familie. **V1-formule (mig 296)**: `voorraad m² − Bruto-maatwerkvraag`. Beantwoordt "hoeveel rol-m² heb ik vandaag écht over voor een **nieuwe** maatwerk-order op deze familie?". `besteld_inkoop` bewust **buiten** de KPI — staat als losse pill ernaast met leverweek (analoog aan mig 149-keuze om IO uit [[Vrije voorraad]] te halen). **Claims (`producten.gereserveerd`) óók buiten de V1-KPI** — die cache is `SUM(aantal)` in **stuks**, niet m² (mig 149); 1-op-1-aftrek zou voor gemengde families fout zijn. V2-backlog: conversie `aantal × stuk_m²` via `producten`-join, dan terug naar `voorraad − Claims − Bruto-maatwerkvraag` (zie ADR-0026 §V2-backlog #6). Zichtbaar als chip op de familie-rij van Rollen & Reststukken, sorteerbaar via dropdown. **V1 = puur inzicht**: geen drempel, geen kleurcodering, geen automatische bestel-trigger. Drempel + alarm-modus + Inkoop-radar-pagina staan op V2-backlog. |
| **Volle rol** | Rol met standaard breedte én volledige lengte. `rol_type = 'volle_rol'`. Standaard breedte komt primair uit `kwaliteiten.standaard_breedte_cm` (bron van waarheid sinds migratie 086), fallback op laatste 3 cijfers artikelnr, daarna 400 cm. |
| **Aangebroken rol** | Rol met standaard breedte maar reeds aangesneden (kortere lengte, ≥100 cm). Ontstaat na `voltooi_snijplan_rol()`. `rol_type = 'aangebroken'`, `status = 'beschikbaar'`. |
| **Reststuk** | Rol met afwijkende (smallere) breedte t.o.v. standaard, óf met lengte <100 cm. `rol_type = 'reststuk'`. Classificatie gebeurt automatisch via trigger o.b.v. `bereken_rol_type()`. Fysiek ontstaan uit `voltooi_snijplan_rol()`/`maak_reststuk()` met eigen rolnummer, gekoppeld aan `oorsprong_rol_id`. Kwalificatie-drempel voor het snijplan-algoritme: korte zijde ≥ 50 cm én lange zijde ≥ 100 cm (`RESTSTUK_MIN_SHORT/LONG` in `_shared/guillotine-packing.ts` + `_shared/compute-reststukken.ts` + `frontend/.../snijplanning/lib/compute-reststukken.ts`). Onder die drempel = snij-afval, niet apart ingestickerd. Boven die drempel = herbruikbaar — maar zie [[Reststuk-scoring]] voor wáár de packer ze laat ontstaan. |
| **Magazijnleeftijd** | Aantal dagen dat een rol fysiek in het magazijn ligt. Bron: `rollen.in_magazijn_sinds` (ADR-0021). Bepaalt het kleurverschil-risico: rollen van dezelfde kwaliteit+kleur zijn vers identiek en lopen alleen door veroudering uiteen. Reststukken/aangebroken rollen **erven** de leeftijd van hun moederrol — de klok reset niet bij snijden. NIET af te lezen uit `reststuk_datum` (traceability-datum die wél bij elke ontvangst/snijbeurt op de huidige datum gezet wordt). |
| **FIFO-snijden** | Voorkeur om de oudste voorraad eerst weg te snijden zodat kleurverschil wordt vermeden. In de snijplanner-packer een **kost-dimensie** naast snijverlies: oudere rollen verlagen de pack-kost (meer snijverlies acceptabel om ze eruit te halen), met absolute voorrang boven de harde bovengrens. Begrensd door C1 (geen verdringing van gereserveerde rollen) en C2 (geen deadline-schade). Tunebaar via `app_config.snijplanning`. Zie ADR-0021. |
| **Standaard rolbreedte** | Per kwaliteit vastgelegd in `kwaliteiten.standaard_breedte_cm`. Voorbeelden: 400 cm (default, CISC/GALA/VERR/…), 320 cm (OASI/NOMA/RUBI/CAVA/EMIR/…), 200 cm (DUBE/VETB/CLAS), 147-180 cm (BUX-serie lopers), 160 cm (BEAC/BEAB), 500 cm (CROW). |
| **Artikelnr-codering** | Veel (niet alle) Karpi-artikelnummers volgen: 4 letters (kwaliteit) + 2 cijfers (kleur) + 3 cijfers (breedte in cm). Voorbeeld: `CISC12400` = CISC kwaliteit, kleur 12, breedte 400 cm. Voor kwaliteiten zonder dit suffix (OASI-serie e.d.) geldt uitsluitend `kwaliteiten.standaard_breedte_cm`. |
| **Voorraadpositie per (Kwaliteit, Kleur)** | Eén domeinconcept dat per `(kwaliteit_code, kleur_code)`-paar antwoord geeft op drie vragen: (1) **eigen voorraad** — wat heb ik vandaag uit eigen rol (volle/aangebroken/reststuk + totaal m²); (2) **uitwisselbare partners** — welke andere paren in dezelfde collectie kunnen dit aanvullen, plus de "beste partner" (top-m²-partner, alleen ingevuld wanneer eigen voorraad = 0); (3) **besteld** — wat komt binnenkort uit inkoop (m, m², orders_count, eerstvolgende leverweek/datum). Bron: SQL-RPC `voorraadposities(p_kwaliteit, p_kleur, p_search)` (mig 179) en frontend-Module [`@/modules/voorraadpositie`](../frontend/src/modules/voorraadpositie/) die als enige seam tussen UI en deze drie sub-bronnen staat. Vervangt op termijn de losse callers van `rollen_uitwissel_voorraad`, `uitwisselbare_partners` en `besteld_per_kwaliteit_kleur`. Kleur-normalisatie (`'15.0' → '15'`) is bron-van-waarheid in zowel SQL (regexp_replace) als TS (`normaliseerKleurcode`). |
| **Gewicht/m²** | Density per kwaliteit in kg per vierkante meter. Bron-van-waarheid: `kwaliteiten.gewicht_per_m2_kg` (mig 184). Drijft alle gewicht-berekeningen voor orderregels en zendingen via de gewicht-resolver-Module. Ingegeven via instellingen-pagina `/instellingen/kwaliteiten` of via eenmalige Excel-import (`import/import_kwaliteit_gewichten.py`). NULL = nog niet ingevuld; producten in deze kwaliteit vallen terug op legacy `producten.gewicht_kg` met flag `gewicht_uit_kwaliteit=false`. |
| **Gewicht-resolver** | SQL-Module die gewicht voor een orderregel of product berekent uit oppervlakte × kwaliteit-density. **Smal publiek interface (drie functies, mig 185):** `gewicht_per_m2_voor_kwaliteit(code)`, `bereken_product_gewicht_kg(artikelnr)`, `bereken_orderregel_gewicht_kg(id)`. **Brede implementatie (intern):** oppervlak-bepaling per producttype, NULL-fallback, trigger-cascade kwaliteit → producten → open order_regels. Alle gewicht-callers gaan hierdoor — geen verspreide `oppervlak × density`-formules meer. |
| **Gewicht-cache** | Verzamelnaam voor de gederiveerde gewicht-velden: `producten.gewicht_kg` (kg per stuk, gederiveerd uit kwaliteit-density × `lengte_cm × breedte_cm`) en `order_regels.gewicht_kg` (snapshot bij regel-aanmaak/-update, herrekend voor open orders bij kwaliteit-mutatie via `trg_kwaliteit_gewicht_recalc` + `trg_product_gewicht_recalc`, mig 185). Beide onderhouden door triggers — niet handmatig editen. Verzonden orders bevroren via `zendingen.totaal_gewicht_kg` (mig 176/177). |
| **Gewicht-uit-kwaliteit-flag** | Boolean `producten.gewicht_uit_kwaliteit` (mig 184). TRUE = `gewicht_kg` gederiveerd uit `kwaliteiten.gewicht_per_m2_kg`. FALSE = legacy waarde uit oude systeem (kwaliteit heeft nog geen gewicht ingevuld, of artikelnr-patroon was niet parsbaar). Maakt migratie-voortgang zichtbaar; data-completing-rapport in `/instellingen/kwaliteiten` filtert hierop. UI toont badge `<GewichtBronBadge>` op product-detail bij FALSE. |
| **Bbox-oppervlak (gewicht)** | Voor maatwerk-vormen (rond/ovaal) wordt het gewicht berekend op het omsluitende rechthoek (lengte × breedte), niet op de werkelijke geometrische oppervlakte. Industrie-standaard voor zowel prijs ([`berekenPrijsOppervlakM2`](../frontend/src/lib/utils/maatwerk-prijs.ts)) als gewicht. Voor rond: `diameter²`. | **Vorm (producten)** | `producten.vorm` ∈ `{rechthoek, rond}` (mig 188). Bepaalt welke formule de gewicht-resolver gebruikt voor catalogus-artikelen. | **Bandkleur (label)** | Master-string voor de afwerking-band die op een maatwerk-tapijt komt — bv. "Piero Taupe 431". Vervangt de pre-mig-194 hardcoded `Piero `-prefix in de UI en de losse `band_merk`/`band_omschrijving`/`band_kleur`-velden in `maatwerk_band_defaults`. **Default per kwaliteit/kleur** zit in `maatwerk_band_defaults.afwerking_kleur_id`, beheerd via /producten kleur-uitvouw. **Order-form** gebruikt strict-FK-dropdown — geen vrije tekst meer. **Bestaande order-regels** behouden hun tekst-snapshot in `order_regels.maatwerk_band_kleur`; nieuwe regels schrijven de FK in `maatwerk_band_kleur_id` én tegelijk de label-string in de tekstkolom. |

Publieke RPC `levertijd_fit_check(p_regel_ids[], p_gewenste_week)` → `{haalbaar, reden?, eerstvolgend_haalbaar?}`. Voor maatwerk-regels = capaciteit-match tegen open snijplannen + `productie_planning`-config (capaciteit_per_week, wisseltijd_minuten, logistieke_buffer_dagen). Voor voorraad-regels = read-only lookup in Reservering's `order_regel_levertijd`-view + uitwisselbaar-dekking — geen eigen rekenpad. Aangeroepen vanuit order-form bij elke regel-mutatie (debounced) én bij commit als finale validatie. |
| **Levertijd-snelste-haalbaar** | "Wat is de snelste leverweek voor deze orderregel(s) als we capaciteit forceren?" — beantwoord door [[Levertijd-Module]] op operator-aanvraag ("Klant heeft haast — toon snelst haalbare"). Publieke RPC `levertijd_snelste_haalbaar(p_regel_ids[])` → `{snelste_haalbaar, spoed_uitleg?}`. Voor maatwerk = mid-week-slot of uitwisselbaar-omsticker-pad. Voor voorraad = onmiddellijk (pick + verzend, 1-2 werkdagen) of eerstvolgende IO via Reservering's claim-info. Operator kan resultaat overnemen — vult `orders.afleverdatum` aan met de gekozen week en Module zet `levertijd_status='eerder_dan_standaard'`. Geen capaciteit-slot-reservering in V1 (race-conditie bij gelijktijdige spoed-aanvragen los je optimistisch op bij commit). |

## Magazijn & Pick-flow

| Term | Betekenis |
|------|-----------|
| **Magazijn** | De Module die de fysieke voorraad-locatie en pickbaarheid van orderregels bezit. Eigenaar van de pick-overview, locatie-mutaties op rollen + snijplannen, en de pick-bucket-categorisering. Frontend-folder: `modules/magazijn/`. Niet te verwarren met `magazijn_locaties` (de tabel) of `Voorraad` (de bezittelijke kant van producten). _Vermijd:_ "pick-ship" als domeinconcept — dat was de oude folder-naam, niet een entiteit. |
| **Pickbaarheid** | Per orderregel: kan deze nu uit het magazijn opgepakt worden? Bron: view `orderregel_pickbaarheid` (samengesteld uit voorraad-claims, snijplan-status, confectie-status, locatie). Een regel is `is_pickbaar=true` als `pickbaar_stuks ≥ totaal_stuks`. Magazijn-Module heeft een fallback achter de seam voor envs zonder de view. |
| **Pick-bucket** | Pick-week-tab op de pick-overview: `wk_1`..`wk_5` (vijf pick-weken — `wk_1` = huidige pick-week + achterstallig, `wk_5` = pick-week + 4) of `later` (zes weken vooruit of geen datum). Pure functies `bucketVoor(afleverdatum, vandaag)` + `genereerWeekTabs(vandaag)`. Tab-label = pick-week (bv. "Week 19"); de bijbehorende verzendweek (1 week later) staat als pill in de sectie-kop ("Verzendweek 20"). Karpi-vuistregel: picken altijd 1 week vóór verzenden. |
| **Wacht-op** | Reden waarom een orderregel niet pickbaar is: `snijden`, `confectie`, `inpak`, of `inkoop`. Komt uit de pickbaarheid-view en wordt op de pick-card als labelled badge getoond. |
| **Magazijn-locatie** | Fysieke opslagplek (rek/vak-code, vrije tekst). Tabel `magazijn_locaties` met `code` (PK-stijl), `omschrijving`, `type`, `actief`. Wordt gekoppeld aan een rol (`rollen.locatie_id`) of via de magazijn-Module aan een afgewerkt maatwerk-stuk (kolom `locatie` op `snijplannen`). RPC `set_locatie_voor_orderregel(p_order_regel_id, p_code)` is de atomic write-pad. |
| **Verzendset** | Magazijn-output bij overhandiging aan logistiek: een set verzendstickers (GS1-128/SSCC) + A4-pakbon, geprint vanuit `/logistiek/:zending_nr/printset`. Gestart door de `<VerzendsetButton>` op een volledig pickbare order — knop leeft in `modules/logistiek/`, niet magazijn, omdat hij `create_zending_voor_order` aanroept. **De klik start tegelijk de Pickronde**: de zending wordt aangemaakt in status `Picken`, niet meer direct in `Klaar voor verzending` (zie ADR-0003). |
| **Vervoerder-tag** | Visuele badge op de pick-card die de geselecteerde vervoerder toont (HST/Rhenus/Verhoek/…). Component leeft in `modules/logistiek/`, geconsumeerd door `OrderPickCard` in `modules/magazijn/` als slot — een seam zonder data-coupling. |
| **Pickronde** | De fysieke pick-bezigheid voor één Zending: tapijten verzamelen, SSCC-stickers plakken, gereedzetten voor de vervoerder. 1-op-1 met `zendingen` (één Pickronde per zending). Eigenaar: **Magazijn-Module** — Pickronde muteert alleen de status van de zending (logistiek-tabel) via een smal RPC-contract, geen eigen tabel. Levenscyclus = `zending_status`-flow: `Picken` (start = klik op Verzendset) → `Klaar voor verzending` (klik "Voltooi pickronde"). De HST-/EDI-trigger op `Klaar voor verzending` (`trg_zending_klaar_voor_verzending`) vuurt nu pas op het echte voltooi-moment. **Picker-claim**: bij start kiest de operator zichzelf uit een dropdown (rol Picker op een Medewerker). Diens id staat op `zendingen.picker_id` en op elke `zending_colli.gepickt_door_id` na voltooien. **Sluitstuk factuur-keten (ADR-0005)**: `voltooi_pickronde` flipt zending naar `Klaar voor verzending` én — als dat de laatste open zending van de order is — `orders.status='Verzonden'`, waardoor de factuur-queue (mig 118) automatisch vuurt. Zie ADR-0003 (Pickronde-introductie), ADR-0004 (Picker-rol), ADR-0005 (factuur-sluitstuk). |
| **Pick-uitkomst** | Per `zending_colli` de uitkomst die de operator tijdens de Pickronde noteert: `open` (default), `gepickt` (sticker geplakt, klaar voor verzending) of `niet_gevonden` (operator kon het tapijt niet vinden). Default-flow: bij "Voltooi pickronde" worden alle `open`-colli's automatisch op `gepickt` gezet — operator hoeft alleen uitzonderingen actief te markeren. |
| **Niet-gevonden-flow** | Wat gebeurt als operator een colli markeert als `niet_gevonden`: dialog vraagt **Blokkeer & escaleer** (zending blijft `Picken`, regel verschijnt op pick-problemen-werklijst voor magazijnchef — voorraad-correctie / andere rol / uitwisselbaar / terug naar inkoop) of **Splits zending** (colli wordt losgekoppeld, blijft open in de order voor een latere Pickronde — alleen geldig bij `lever_modus='deelleveringen'`). Operator beslist per geval. |

## Facturatie

| Term | Betekenis |
|------|-----------|
| **Facturatie-Module** | De Module die de factuur-flow vanaf de wekelijkse cron tot bezorgde PDF/EDI-INVOIC bezit. Frontend-folder: `modules/facturatie/` met eigen pages, components, hooks, queries — incl. de queries voor `klant.btw_percentage` en `klant.email_factuur` (al staan die velden op de debiteuren-rij). Edge functions `factuur-verzenden` en `factuur-pdf` zijn fysiek in `supabase/functions/` maar mentaal eigendom van deze Module. Sinds [ADR-0010](adr/0010-factuur-volgt-bundel-zending.md) volgt elke factuur één bundel-zending — geen status-overgang-trigger meer; `enqueue_wekelijkse_verzamelfacturen` itereert maandagochtend over bundel-zendingen van vorige week. Cross-cuts: `<FactuurOrderSlot>` (in components/orders) en `<KlantFactuurInstellingen>` (in components/klanten) blijven bij hun host-folder maar consumeren de Module-barrel. Zie [ADR-0007](adr/0007-facturatie-als-deep-module.md) en [ADR-0010](adr/0010-factuur-volgt-bundel-zending.md). |
| **Bundel-factuur** | Eén factuur per bundel-zending. Aggregatie-eenheid is de 4-dim bundel-sleutel uit mig 228: `(debiteur × adres-norm × vervoerder × verzendweek)`. Een klant met 2 verschillende afleveradressen of 2 verschillende vervoerders in dezelfde week krijgt N facturen — één per bundel = één per pakbon = één per fysieke transportbeweging. Drempel-toets (`debiteuren.verzend_drempel`, `gratis_verzending`) gebeurt op het bundel-totaal, zodat 2× €300 in dezelfde bundel boven een €500-drempel uitkomt en de klant geen verzendkosten betaalt. Bron: RPC `genereer_factuur_voor_bundel(p_zending_id)` (mig 235). Vervangt de mig 232-aggregatie op `(debiteur, week)`. Zie [ADR-0010](adr/0010-factuur-volgt-bundel-zending.md). |
| **Bundelkorting** | Service-korting die we toepassen wanneer een bundel-zending (mig 222) tegelijk de verzendkosten-drempel van de klant overschrijdt. Op factuur 2 regels: volle VERZEND + tegenboeking BUNDELKORTING (artikelnr-conventie sinds mig 256). Saldo verzendkosten voor de klant: € 0. Communiceert het cadeau zichtbaar; alleen 1 regel met € 0 zou de service verbergen. |
| **factuurvoorkeur** | ~~Per-klant veld `debiteuren.factuurvoorkeur` (`per_zending` | `wekelijks`).~~ **Vervallen per [ADR-0010](adr/0010-factuur-volgt-bundel-zending.md)** (2026-05-08): kolom gedropt in mig 234. `per_zending` werkte tegen de bundel-drempel-strategie en was de-facto inactief; `wekelijks` is voortaan impliciet en de enige modus. UI-radio in klant-detail-tab is verwijderd. |
| **factuur_queue** | Append-only queue-tabel die jobs voor de `factuur-verzenden` edge function bevat. Statussen `pending` → `processing` → `done` / `failed`. Eén rij per bundel-factuur (FK `zending_id` naar de bundel-zending; `order_ids[]` afgeleid uit `zending_orders` bij enqueue). Recovery via `recover_stuck_factuur_queue()` (mig 121, elke 5 min): rijen >10 min in `processing` worden teruggezet op `pending`. pg_cron-drain elke minuut (mig 122). Atomic claim via `claim_factuur_queue_items` (mig 227, `FOR UPDATE SKIP LOCKED`). |
| **Verzendkosten-resolver** | SQL-functie `verzendkosten_voor_bundel(p_debiteur_nr, p_bundel_subtotaal, p_is_afhalen)` (mig 235) die de 4-paden-toets concentreert: `gratis_afhalen`, `gratis_klantafspraak` (klant.gratis_verzending=TRUE), `gratis_drempel` (subtotaal ≥ klant.verzend_drempel), of `betaald` (klant.verzendkosten). Returnt `(te_betalen NUMERIC, status TEXT, reden TEXT)`. View `voorgestelde_zending_bundels` (mig 229) en `genereer_factuur_voor_bundel` consumeren beide deze functie — single source of truth voor de drempel-logica. Vóór mig 234 leefde de toets in 4 verspreide CASE-takken. |
| **BTW verlegd (intracommunautair)** | Bij B2B-levering aan een afnemer in een ander EU-land wordt de BTW "verlegd" naar de afnemer: Karpi factureert 0% en vermeldt "BTW verlegd" + het btw-nummer van de afnemer op de factuur; de klant draagt zelf BTW af in eigen land. Vlag: `debiteuren.btw_verlegd_intracom`; snapshot per factuur: `facturen.btw_verlegd` (mig 371). Vereist formeel een geldig btw-nummer van de afnemer (`debiteuren.btw_nummer`) — ontbreekt dat, dan waarschuwt de klant-facturering-tab maar blokkeert niets. |

## Voorraad & Magazijn

### Voorraadcorrectie (handmatige rol-mutatie)
Handmatig toevoegen/bewerken/verwijderen van een rol of reststuk op de Rollen &
Reststukken-pagina, voor inventarisatie of het rechtzetten van telfouten,
historische rollen, beginvoorraad of fysiek verlies. Loopt via de RPC's
`rol_handmatig_toevoegen` / `_bewerken` / `rol_verwijderen` (mig 291-293) en
wordt gelogd in `rol_mutaties` met verplichte reden. Raakt `producten.voorraad`
bewust niet (ADR-0024).

### Migratie-blokkering
**Migratie-blokkering** — eenmalige FIFO-lengtereservering van een nog-niet-gesneden oud-systeem maatwerk-order op een fysieke rol, full-width (volle rolbreedte × min(A,B) cm). Voorkomt dat de overgenomen voorraad dubbel wordt verkocht. Vervalt zodra de order gesneden is (status `vrijgegeven`). Zie ADR-0028 (mig 313–314).
