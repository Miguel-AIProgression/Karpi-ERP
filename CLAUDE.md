# RugFlow ERP — Karpi Tapijtgroothandel

ERP-portaal voor Karpi: beheer van klanten (debiteuren), producten (tapijtrollen), orders, en operationele processen. Gebouwd op Supabase + React/TypeScript.

## Tech Stack
- **Backend:** Supabase (PostgreSQL, Auth, Storage, Edge Functions)
- **Frontend:** React 18+, TypeScript, Vite, TailwindCSS, shadcn/ui, TanStack Query
- **Import:** Python 3 (pandas, openpyxl, supabase-py)

## Database kernconcepten
- 37 tabellen, 7 enums, 14 views, 24 functies — zie `docs/database-schema.md`
- `debiteur_nr` (INTEGER) = PK voor klanten — alle brondata en logo's verwijzen hiernaar
- `artikelnr` (TEXT) = PK voor producten
- Kwaliteitscodes (3-4 letters) → collecties (groepen uitwisselbare kwaliteiten)
- Orders bevatten adres-snapshots (geen FK naar afleveradressen)
- Nummering: `volgend_nummer('ORD')` → ORD-2026-0001

## Bedrijfsregels
- **Orders verwijderen:** mag alleen als er geen snijplannen zijn met status 'Snijden', 'Gesneden' of later. Snijplannen met vroege statussen ('Wacht', 'Gepland') moeten eerst handmatig verwijderd worden vóór de order. Gooit anders FK-fout op `snijplannen_order_regel_id_fkey`.
- **Orders annuleren cascadeert naar Snijplanning (ADR-0023, mig 290):** `markeer_geannuleerd` (mig 218) schrijft een `geannuleerd`-event in `order_events`; daarop reageren twee ontkoppelde listeners — `trg_order_events_reservering_release` (mig 255, releaset voorraad+IO-claims) én `trg_order_events_snijplan_release` (mig 290). De tweede zet **alle** snijplannen van de order op `Geannuleerd` (ongeacht voortgang — bewuste werkvloer-keuze, anders dan bij _verwijderen_) en geeft geraakte rollen die hun laatste actieve (`Gepland/Snijden/Gesneden`) snijplan verliezen vrij naar `beschikbaar`/`reststuk` (`snijden_gestart_op=NULL`), met `NOT EXISTS`-guard zodat een rol die nog een andere order bedient onaangeroerd blijft. Defense-in-depth: `snijplanning_overzicht` filtert sinds mig 290 `WHERE o.status <> 'Geannuleerd'` (bewust NIET ook `'Verzonden'` — die view voedt ook de fysieke rol-uitvoer + packer). Nieuwe `geannuleerd`-cascade-effecten = aanhaken op `order_events` (ADR-0006/0015-patroon), géén edit in `markeer_geannuleerd`.
- **Confectie-planning:** bron is `snijplannen` via view `confectie_planning_forward`. Lane (type_bewerking) wordt afgeleid van `maatwerk_afwerking` via `afwerking_types.type_bewerking`. Afwerkingen `ON`/`ZO` hebben geen lane en verschijnen onder "alleen stickeren". Status-transities lopen via RPC's `start_confectie` en `voltooi_confectie`, niet directe UPDATE.
- **Reservering op inkoop (vaste maten, mig 144–152):** orderregels zonder voldoende voorraad alloceren automatisch op openstaande inkooporderregels (status `Besteld`/`Deels ontvangen`, `eenheid='stuks'`) op `verwacht_datum ASC`. Bron-van-waarheid: tabel `order_reserveringen` met rijen voor zowel voorraad- als IO-claims. Centrale RPC: `herallocateer_orderregel(id)` — idempotent, wordt vanuit triggers en handmatig aangeroepen. **Claim-volgorde-prio**: wie eerst claimt, wordt eerst beleverd; geen automatische herallocatie bij urgentere nieuwe orders — **met één gerichte uitzondering: claim-swap (ADR-0027)**. Wanneer een nieuwe urgente orderregel voorraad-tekort heeft, mag de allocator een voorraad-claim van een andere order verplaatsen naar een IO, mits (1) die andere order's `afleverdatum > standaard_afleverdatum_berekend` (= operator heeft bewust later ingevuld) én (2) er een IO bestaat met `verwacht_datum + inkoop_buffer_weken_vast ≤ A.afleverdatum`. Selectie bij meerdere kandidaten: **EDD** (meeste headroom verliest); IO-keuze: **laatst-passend** (bewaart vroege IO's voor toekomstige urgent). Eénstaps, geen cascade. Audit-trail via `order_events`-rij `'claim_geswapt'`. Géén nieuwe kolommen — hergebruikt bestaande `afleverdatum` en `standaard_afleverdatum_berekend` (ADR-0020-snapshot). Bij IO-annulering schuiven claims naar volgende IO of orders gaan in "Wacht op nieuwe inkoop" (orderregel zonder dekking — order-status blijft `Wacht op inkoop` totdat een nieuwe IO de allocator opnieuw triggert via orderregel-bewerking).
- **Maatwerk levertijd-indicator:** maatwerk reserveert NIET op inkoop in V1. Op de orderregel verschijnt alleen een hint `Eerstvolgende inkoop wk + 2 weken` als er geen rol beschikbaar is. Echte claim op rol-IO (`eenheid='m'`) staat op de V2-backlog.
- **FIFO-magazijnleeftijd in de snijplanner (ADR-0021, mig 280-286):** kleurverschil tussen rollen van dezelfde kwaliteit+kleur ontstaat puur door fysieke veroudering. Bron-van-waarheid is `rollen.in_magazijn_sinds` (NIET `reststuk_datum` — dat reset bij snijden en wordt ook voor volle IO-rollen op `NOW()` gezet). Reststukken/aangebroken rollen **erven** `in_magazijn_sinds` van de moederrol. **Modus-schakelaar `app_config.snijplanning.modus` (mig 285, default `'simpel'`):** in `simpel` (huidig live) pakt de packer strikt de oudst-binnengekomen rol eerst — géén kost-afweging/badge/carve-out (`fifoMetrics` leeg); de rollen-pagina toont per (kw,kl) binnenkomstdatum + "1e binnen" (mig 286, `voorraadposities` sorteert oudste-eerst). In `geavanceerd` weegt de packer (`_shared/guillotine-packing.ts`) leeftijd als kost: `afval − α·max(0, leeftijd − drempel)`, drempel 90 dgn, absolute voorrang ≥180 dgn, tunebaar via `app_config.snijplanning`; **C1** geen verdringing van rollen in een goedgekeurd voorstel; **C2** geen stuk met afleverdatum laten vallen dat efficiency wél plaatste; short-circuit bij verse voorraad; **rode** badge → in `auto-plan-groep` NIET auto-goedgekeurd. Zonder `PackOptions.fifo` is het packer-gedrag exact als voorheen. Omschakelen naar `geavanceerd` zodra de interne rol-data op orde is.
- **`lever_modus`:** order-niveau keuze "deelleveringen" / "in_een_keer". Default uit `debiteuren.deelleveringen_toegestaan`, gevuld via `LeverModusDialog` bij opslaan als ≥1 regel tekort heeft. Bepaalt levertijd-berekening (eerste IO-week resp. max IO-week) en aantal zendingen. NULL voor orders zonder tekort.
- **`lever_type` (ADR 0014 / mig 244):** order-niveau intentie `'week'` (B2B-default, ~90%) of `'datum'` (B2C). Dag-orders verschijnen pas 1 werkdag vóór afleverdatum in Pick & Ship (helper `werkdagMinN` in de werkagenda-kernel [`_shared/werkagenda.ts`](supabase/functions/_shared/werkagenda.ts); frontend importeert die direct via `bereken-agenda.ts`; horizon + snij-deadline tellen met de centrale werkagenda-config (`app_config 'werkagenda'`, mig 384 — feestdagen/werkdagen)) en krijgen voorrang in [`check-levertijd`](supabase/functions/check-levertijd/index.ts) (kritieke deadline = afleverdatum − `app_config.productie_planning.dag_order_snij_buffer_werkdagen`, default 2 werkdagen). Default per klant in `debiteuren.default_lever_type`; per order overschrijfbaar via segmented toggle in [`LeverDatumField`](frontend/src/components/orders/order-form.tsx). Bundel-sleutel (mig 228-230), wekelijkse factuur-cron (mig 231-232) en IO-sync (mig 153) ongewijzigd — dag- en week-orders bundelen normaal als de 4D-sleutel matcht. Visuele badges (terracotta 📅) op order-detail, Pick & Ship en orders-overzicht onderscheiden dag-orders.
- **`vrije_voorraad`-formule (mig 149):** `voorraad − gereserveerd − backorder` (geen `+ besteld_inkoop` meer). `gereserveerd` is voortaan SUM van actieve `bron='voorraad'`-claims uit `order_reserveringen`, niet meer SUM van `te_leveren` op order_regels. Toekomstige inkoop blijft zichtbaar via `besteld_inkoop` en de claims, maar telt niet meer in "vandaag-leverbaar".
- **Admin-pseudo-orderregel (ADR-0018, mig 272-273):** orderregels waarvan het artikel administratief is (geen fysieke leverbaarheid) worden uniform geskipt in allocator, status-bepaling, levertijd-view, pickbaarheid en dekking-preview. Bron-van-waarheid: `producten.is_pseudo BOOLEAN` (mig 272). Predikaten: SQL `is_admin_pseudo(artikelnr)` (mig 272) en TS `isAdminPseudo(regel)` ([`@/lib/orders/admin-pseudo`](frontend/src/lib/orders/admin-pseudo.ts) — accepteert zowel form-data met `is_pseudo` top-level als query-resultaten met `producten ( is_pseudo )`-join). Nieuwe admin-pseudo toevoegen = pure `UPDATE producten SET is_pseudo=TRUE` — geen code-edit. **Niet te verwarren** met `SHIPPING_PRODUCT_ID='VERZEND'`: die constant bedient toe-voegen van een verzendregel (`applyShippingLogic`), niet skip-detectie. De oorspronkelijke driedubbele-fix (mig 263→266→269) en de claim-keten-recursie-bug (mig 263→267) komen categorisch niet meer voor — er is geen string-lijst om uit te breiden.
- **Afleverdatum sync met IO-claims (mig 153):** `herwaardeer_order_status` synct na elke alloc-cyclus de `orders.afleverdatum` naar de laatste IO-claim-leverdatum (`MAX(verwacht_datum) + inkoop_buffer_weken_vast × 7` dagen) als die later valt dan de huidige afleverdatum. Schuift alleen vóóruit, nooit terug — de afleverdatum reflecteert dus altijd minimaal de werkelijke claim-belofte aan de klant. Eindstatussen (Verzonden / Geannuleerd / Klaar voor verzending) blijven ongewijzigd.
- **Uitwisselbaar = handmatige claims (mig 154):** orderregel kan multi-source gedekt worden — voorraad eigen artikel + voorraad uitwisselbare(n) (omgestickerd) + IO eigen artikel — zonder de regel zelf te splitsen. Gebruiker kiest in [`UitwisselbaarTekortHint`](frontend/src/components/orders/uitwisselbaar-tekort-hint.tsx) per uitwisselbaar product een **aantal**; bij submit roept `order-form` `set_uitwisselbaar_claims(regel_id, [{artikelnr, aantal}])` aan die `is_handmatig=true` claims aanmaakt. Allocator (`herallocateer_orderregel`) **respecteert handmatige claims** (releaset alleen niet-handmatige) en vult resterend aan met voorraad eigen + IO eigen. **Voorraad uitwisselbaar product wordt ook daadwerkelijk gereserveerd** — `herbereken_product_reservering` telt op `fysiek_artikelnr`. **Factuur en order-regel weergave blijven 1× origineel artikel**; de omstickering is intern. Op order-detail wordt de claim-uitsplitsing per stuks-regel zichtbaar als geneste sub-rijen (eigen voorraad → omsticker → IO → wacht op nieuwe inkoop), met locatie + omschrijving van het uitwisselbare bron-product bij omsticker-rijen — gericht op de verzamelaar in het magazijn. Bron: [`fetchClaimsVoorOrder`](frontend/src/lib/supabase/queries/reserveringen.ts) + hook `useClaimsVoorOrder`.
- **Inkoopgroepen (mig 189):** 10 INKC-organisaties (BEGROS, DECOR UNION, FACHHANDELSRING, INTERRING, VME, VME (TH), TINTTO, INHOUSE, HOUSE OF DUTCHZ, MUSTERRING) staan in tabel `inkoopgroepen`. Een debiteur hoort aan **maximaal 1** inkoopgroep via FK `debiteuren.inkoopgroep_code`. `orders.inkooporganisatie` blijft TEXT-snapshot — orders bewegen niet mee bij wijziging op debiteur. Centraal beheer onder `/inkoopgroepen`; klant-detail toont de groep als link.
- **Pick & Ship pickbaarheidsfilter:** een order verschijnt pas in Pick & Ship zodra **alle** regels gepickt kunnen worden. Reden voor onpickbaar maakt niet uit: 'Wacht op snijden', 'Wacht op inkoop', 'Wacht op confectie/inpak', of géén regels (header-only webshop-orders). Uitzondering: klanten met `debiteuren.deelleveringen_toegestaan=true` zien een gemixte order al wél zodra ≥1 regel pickbaar is — operator stuurt dan een deellevering. Orders zonder enkele pickbare regel verdwijnen ook bij deelleveringen. Gefilterd in [`fetchPickShipOrders`](frontend/src/modules/magazijn/queries/pickbaarheid.ts); stats lopen via dezelfde query. **Regels worden gechunkt per order_id opgehaald** (fix 2026-06-11): een kale GET op `orderregel_pickbaarheid` liep tegen de PostgREST max-rows-cap (1000) waardoor orders boven die grens stilletjes uit de lijst vielen (91 van ~236 zichtbaar). **Geen vervoerder ≠ onzichtbaar:** orders met ≥1 regel `bron='geen'` staan gewoon in de lijst, maar pick-start is dubbel geblokkeerd met "Geen vervoerder mogelijk" — disabled knop in [`StartPickrondesButton`](frontend/src/modules/logistiek/components/start-pickrondes-button.tsx) én guard in `start_pickronden` (mig 373). Escape-hatch: vervoerder-override op de orderregel.
- **Zending-bundeling op afleveradres (mig 222):** orders met identiek genormaliseerd afleveradres (postcode + adres + land, alle whitespace genormaliseerd, uppercase via `_normaliseer_afleveradres`) én dezelfde effectieve vervoerder, binnen dezelfde debiteur, worden bij het starten van een Pickronde automatisch gebundeld in 1 zending → 1 pakbon → 1 transportorder. Bron-van-waarheid voor de orders-per-zending is de M2M-tabel `zending_orders` (backfill heeft 1-op-1 zendingen ook gevuld zodat queries uniform werken). RPC: `start_pickronden_bundel(order_ids[], picker_id)` — valideert de invarianten, groepeert orderregels van álle orders op effectieve vervoerder uit mig 219 en maakt 1 zending per vervoerder-groep, gekoppeld aan alle betrokken orders. Bij 1 order delegeert naar `start_pickronden_voor_order` (mig 220) zodat callers één code-pad hebben. `voltooi_pickronde` is bundel-aware: leest betrokken orders uit `zending_orders` en flipt élke order naar Verzonden zodra dit de laatste open zending is. UI: `BulkVerzendsetButton` clustert vóór de RPC-aanroep zodat de operator vooraf in de popover ziet hoeveel adres-bundels gemaakt worden; pakbon-document toont per bron-order een sub-kop met regels onder elkaar zodat magazijnier én ontvanger zien welke regel bij welke orderbevestiging hoort.
- **Dynamische bundel-preview + week-dimensie (mig 228-230):** bundel-sleutel is **4-dimensionaal** — `(debiteur × adres-norm × effectieve vervoerder × verzendweek)`. Helper-functies `bundel_sleutel(...)` + `verzendweek_voor_datum(date)` (ISO-week `YYYY-Www`) zijn de single source of truth, gespiegeld in [`bundel-sleutel.ts`](frontend/src/lib/orders/bundel-sleutel.ts). Pure SQL-view `voorgestelde_zending_bundels` (mig 229) levert de live preview vóór pickronde-start: geen state, geen materialized view — herevalueert per query. Frontend cachet via React Query (`['voorgestelde-bundels']`, staleTime 60s) en invalidate't bij vervoerder-override, afleverdatum-mutatie en pickronde-start. Pick & Ship `KlantClusterBlok` clustert óók op de 4D-sleutel (niet op debiteur) zodat orders met verschillende vervoerders binnen dezelfde klant in losse cards staan; per échte bundel verschijnt een truck-icoon + adres-snippet + besparing-badge (geen drempel-progressbar — dat is factuur-info, niet picker-info). `start_pickronden_bundel` valideert nu ook identieke verzendweek; `zendingen.verzendweek` is een snapshot. Trigger `trg_lock_zending_bundel_sleutel` (mig 230) blokkeert mutatie van afleverdatum/afl_*/debiteur op orders die in een actieve bundel-zending zitten (`Klaar voor verzending`+).
- **Wekelijkse verzamelfactuur (mig 231-232):** debiteuren met `factuurvoorkeur='wekelijks'` (mig 117) krijgen op maandag 05:00 UTC één factuur per `(debiteur, ISO-week)`. Cron `enqueue_wekelijkse_verzamelfacturen` (mig 122 + 231) groepeert orders waarvan `verzendweek_voor_datum(afleverdatum)` = vorige week, met dubbele-vuur-bescherming via `NOT EXISTS`-check op pending/processing/done queue-rijen. RPC `genereer_factuur_voor_week(debiteur_nr, jaar_week)` (mig 232) volgt mig 227 no-op-guard pattern en voegt **per bundel-zending** van die week één VERZEND-factuurregel toe. Drempel-toets per bundel: `gratis_verzending=TRUE` op de debiteur, of bundel-subtotaal ≥ `verzend_drempel` → bedrag = 0. Een bundel = 1 fysieke transportbeweging — twee vervoerders in dezelfde week = 2 verzendkosten-regels (mits onder drempel). Edge function [`factuur-verzenden`](supabase/functions/factuur-verzenden/index.ts) splitst op `item.type`: 'wekelijks' → `genereer_factuur_voor_week`; 'per_zending' → ongewijzigd `genereer_factuur` (drempel-logica voor per_zending staat op V2-backlog). **EDI-mail-gate (2026-06-11):** `factuur_uit && transus_actief` → uitsluitend EDI-INVOIC, géén factuur-e-mail (mail-gate `!ediFactuurActief` in `factuur-verzenden`; `verstuurd_naar='EDI Transus'`).
- **EDI/Transus-koppeling (mig 156–157, fase 1):** vervangt Windows Connect op MITS-CA-01-009. Alle EDI-verkeer gaat via centrale tabel `edi_berichten` (audit + queue) gekoppeld aan Transus' SOAP-API (M10100 versturen, M10110 poll-ontvangen, M10300 ack). Format: fixed-width "Custom ERP" (Transus-Online ID 17653, versie 10) — Transus vertaalt zelf naar/van EDIFACT D96A richting de partners. Karpi-GLN: `8715954999998`. Per-debiteur configuratie staat in `edi_handelspartner_config` (4 berichttype-toggles + `transus_actief` + `test_modus`). **Cutover-constraint** (uit Transus' antwoord): WC en API kunnen niet parallel voor dezelfde partner — cutover is dus big-bang voor alle 39 partners. **Fase 1 (mig 156–157):** inkomende orders alleen loggen + acken (geen order-creatie), uitgaande berichten nog niet actief — eerst rondreis-test met Transus' test-handelspartner. Plan: [`docs/superpowers/plans/2026-04-29-edi-transus-koppeling.md`](docs/superpowers/plans/2026-04-29-edi-transus-koppeling.md).
- **EDI centrale-facturatie + filiaal-mapping (mig 306):** sommige ketens (Hornbach) factureren centraal maar leveren op filiaal. De inkomende order heeft dan gefactureerd-GLN = (vaak **inactieve**) hoofd-AG en besteller/aflever-GLN = de **fysieke vestiging**. `matchDebiteur` ([`transus-poll/index.ts`](supabase/functions/transus-poll/index.ts)) matcht daarom **meest-specifiek-eerst**: aflever-GLN → `afleveradressen.gln_afleveradres`, besteller-GLN → afleveradres, besteller/gefactureerd-GLN → `debiteuren.gln_bedrijf` (**inactieve debiteuren overgeslagen** zodat Hornbach op de actieve NL **361208** landt, niet op AG **361214**). Matching is tolerant voor het `.0`-import-artefact (matcht `gln` én `gln.0`). Onbekende vestiging-GLN's worden **éénmalig handmatig** gekoppeld via RPC `koppel_edi_afleveradres(p_bericht_id, p_debiteur_nr, p_afleveradres_id)` (de gele koppel-widget op bericht-detail, [`koppel-vestiging-widget.tsx`](frontend/src/modules/edi/components/koppel-vestiging-widget.tsx)): die schrijft de aflever-GLN naar het gekozen afleveradres (= **onthouden**, guard tegen dubbel-koppelen), zet `edi_berichten.debiteur_nr` en roept `create_edi_order` aan. Vanaf de volgende order matcht de vestiging automatisch op stap 1. Overzicht-filter/-badge **"Te koppelen"** = `richting='in' AND berichttype='order' AND order_id IS NULL` (filtert op order_id, NIET op status — de poll laat de status soms op `Verwerkt` staan terwijl creatie faalde). `create_edi_order` (mig 166) matcht het afleveradres al op `gln_afleveradres` en zet de adres-snapshot — geen wijziging nodig daar. **Safety-net op orders-overzicht (2026-06-04):** dezelfde "te koppelen"-definitie voedt via `useTeKoppelenEdiCount()` een rose waarschuwingsbanner ([`EdiTeKoppelenBanner`](frontend/src/modules/edi/components/te-koppelen-banner.tsx)) bovenaan [`orders-overview.tsx`](frontend/src/pages/orders/orders-overview.tsx) — een ongematchte inkomende order mag nooit alleen in de EDI-module hangen. "Koppel nu" deep-linkt naar `/edi/berichten?teKoppelen=1` (filter direct aan). Eén bron-van-waarheid: `frontend/src/modules/edi/lib/te-koppelen.ts` (`isTeKoppelen` + `filterTeKoppelen`); `countTeKoppelenEdiOrders` en de berichten-overzicht-filter consumeren die.
- **EDI debiteur-GLN-alias / meerdere factuur-GLN's per debiteur (mig 307):** centrale-facturatie-ketens kunnen **meerdere factuur-entiteiten** onder één debiteur hebben. ## Werkwijze met Claude Code (team-tips van Boris Cherny)
Deze werkafspraken verhogen kwaliteit en snelheid. Geen dogma's — experimenteer en behoud wat werkt.

### Planning & uitvoering
- **Verticaal implementeren als standaard:** bouw features waar mogelijk in dunne verticale slices — één samenhangend stukje functionaliteit end-to-end door alle lagen (DB/migratie → RPC/edge function → query → UI), zodat het meteen werkt en testbaar is. NIET horizontaal (eerst alle DB, dan alle backend, dan alle frontend). Elke slice levert werkende, demonstreerbare waarde op; pas als verticaal echt niet kan, val terug op een horizontale aanpak.
- **Plan-mode eerst bij niet-triviale taken:** maak eerst een solide plan, laat daarna implementeren. Bij complexe features: één Claude schrijft plan, een tweede reviewt als "staff engineer". Loopt het mis → terug naar plan-mode, niet doorduwen.
- **Detailleer specs:** verwijder ambiguïteit vóór overdracht. Specificiteit = autonomie.
- **Daag output uit:** vraag Claude om keuzes te rechtvaardigen ("prove this works", vergelijk main vs branch). Middelmatige fix? Vraag om verse herschrijving ("scrap it, implement the elegant solution").

### Git-workflow (branch-strategie)
- **Grotere wijzigingen krijgen automatisch een eigen branch.** Begin substantieel werk (nieuwe feature, refactor over meerdere bestanden, migratie, alles met >~1 logische stap of meerdere bestanden) zónder te vragen op een eigen branch — bv. `feat/<korte-naam>`, `refactor/<korte-naam>`, `fix/<korte-naam>`. Commit het werk daar, en **merge pas naar `main` wanneer ik dat expliciet zeg** ("merge maar", "naar main"). Niet automatisch mergen.
- **Kleine, triviale wijzigingen** (één-regel-fix, typo, docs-tweak) mogen direct op `main` — gebruik je oordeel; bij twijfel: aparte branch.
- **Reden:** er draaien vaak meerdere Claude-sessies tegelijk in deze working tree. Direct op `main` werken laat sessies elkaars ongecommitte werk overschrijven/oppakken (zie incident 2026-06-07: gedeelde `changelog.md`/`architectuur.md` raakten verstrengeld). Een eigen branch per substantiële taak isoleert het werk tot ik bewust merge.
- **Merge-moment:** bij "merge naar main" → `git checkout main && git pull --ff-only && git merge <branch>` (of fast-forward), daarna push. Los merge-conflicten op de branch op, niet op `main`.
- Géén PR's nodig (tenzij ik erom vraag) — branch + merge-op-commando volstaat.

### Parallelliseren met git worktrees
- Bij 2+ onafhankelijke taken: gebruik git worktrees zodat elke sessie een eigen werkdirectory heeft.
- Eventueel een vaste "analyse"-worktree voor log/data-onderzoek zonder de hoofdbranch te vervuilen.
