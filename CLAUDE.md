# RugFlow ERP — Karpi Tapijtgroothandel

ERP-portaal voor Karpi: beheer van klanten (debiteuren), producten (tapijtrollen), orders, en operationele processen. Gebouwd op Supabase + React/TypeScript.

## Tech Stack
- **Backend:** Supabase (PostgreSQL, Auth, Storage, Edge Functions)
- **Frontend:** React 18+, TypeScript, Vite, TailwindCSS, shadcn/ui, TanStack Query
- **Import:** Python 3 (pandas, openpyxl, supabase-py)

## Database kernconcepten
- 37 tabellen, 7 enums, 14 views, 24 functies — zie `docs/database-schema.md`
- `debiteur_nr` (INTEGER) = PK voor klanten — alle brondata en logo's verwijzen hiernaar
- `artikelnr` (TEXT) = PK voor producten
- Kwaliteitscodes (3-4 letters) → collecties (groepen uitwisselbare kwaliteiten)
- Orders bevatten adres-snapshots (geen FK naar afleveradressen)
- Nummering: `volgend_nummer('ORD')` → ORD-2026-0001

## Bedrijfsregels
- **Universele Concept-intake-gate (mig 540-542, 2026-06-30/07-01):** elke nieuw aangemaakte order belandt altijd eerst in `order_status = 'Concept'` — ongeacht het intake-kanaal. **Intake-functies (mig 542):** `create_order_with_lines` (handmatig), `create_edi_order` (EDI/Transus) en `create_webshop_order` (Shopify/webshop) plaatsen de INSERT direct op `'Concept'`. `poll-email-orders` gaf al expliciet `'Concept'` mee. **Lekken gedicht (mig 540):** `derive_wacht_status` (no-touch), `herallocateer_orderregel` (guard), `auto_maak_snijplan`/`auto_sync_snijplan_maten` (guards), `actieve_snijgroepen` (NOT IN uitgebreid) — alle vier inert voor Concept-orders. **Enige uitweg: `bevestig_concept_order(p_order_id)` (mig 541)** — doet in volgorde: (1) `_apply_transitie` → 'Klaar voor picken' + audit-event `'concept_bevestigd'`; (2) maatwerk-snijplannen aanmaken (trigger was geblokkeerd); (3) `herallocateer_orderregel_auto` per regel (volledige cascade); (4) `herbereken_wacht_status` voor definitieve status. **Niet te wijzigen:** `registreer_achteraf_order` (mig 524) en `markeer_achteraf_verzonden` (mig 539) slaan de normale workflow bewust over — die blijven ongewijzigd. **TS-spiegel:** `'Concept'` in `EINDSTATUS_OF_PICKRONDE` (`derive-status.ts`), golden fixture 26 cases (was 23).
- **Order aanmaken vereist een gekoppelde prijslijst (mig 481, 2026-06-23):** `create_order_with_lines` (de enige RPC achter de handmatige order-aanmaak-UI) weigert nu een nieuwe order als `debiteuren.prijslijst_nr IS NULL` — `RAISE EXCEPTION` vóór de `orders`-INSERT, plus een frontend-spiegel in `order-form.tsx`'s `saveMutation` (zelfde `throw new Error(...)`-patroon als de afleveradres-gate, mig 392/395) voor directe feedback zonder DB-round-trip. **Aanleiding:** debiteur HEADLAM B.V. (#500001) had nooit een prijslijst-koppeling; 12 open orders stonden daardoor op een te lage prijs (eenmalige datacorrectie +€1.271,43 ex BTW, zie changelog 2026-06-23 — geen code, pure data-fix). **Scope bewust beperkt tot de handmatige creatie-RPC** — `create_edi_order`/`create_webshop_order` blijven ongewijzigd (hebben al een intentionele fallback op `producten.verkoopprijs`, mig 166; een automatisch inkomend kanaal mag niet blijven liggen op een ontbrekende koppeling, de prijs-ontbreekt-gate (mig 396) vangt een verkeerde prijs daar al af). **Alleen creatie, geen edit** — een bestaande order blijft altijd bewerkbaar, ook als de koppeling later ontbreekt. Blast-radius vooraf gecheckt: van de 138 actieve debiteuren zonder prijslijst kwamen de laatste 14 dagen alleen orders binnen via handmatig/`oud_systeem`, nul via EDI/Shopify/webshop — de guard raakt dus geen actieve geautomatiseerde intake.
- **Orders verwijderen:** mag alleen als er geen snijplannen zijn met status 'Snijden', 'Gesneden' of later. Snijplannen met vroege statussen ('Wacht', 'Gepland') moeten eerst handmatig verwijderd worden vóór de order. Gooit anders FK-fout op `snijplannen_order_regel_id_fkey`.
- **Orders annuleren cascadeert naar Snijplanning (ADR-0023, mig 290):** `markeer_geannuleerd` (mig 218) schrijft een `geannuleerd`-event in `order_events`; daarop reageren drie ontkoppelde listeners — `trg_order_events_reservering_release` (mig 255, releaset voorraad+IO-claims), `trg_order_events_snijplan_release` (mig 290) en `trg_order_events_zending_release` (mig 480, zie hieronder). De tweede zet **alle** snijplannen van de order op `Geannuleerd` (ongeacht voortgang — bewuste werkvloer-keuze, anders dan bij _verwijderen_) en geeft geraakte rollen die hun laatste actieve (`Gepland/Snijden/Gesneden`) snijplan verliezen vrij naar `beschikbaar`/`reststuk` (`snijden_gestart_op=NULL`), met `NOT EXISTS`-guard zodat een rol die nog een andere order bedient onaangeroerd blijft. Defense-in-depth: `snijplanning_overzicht` filtert sinds mig 290 `WHERE o.status <> 'Geannuleerd'` (bewust NIET ook `'Verzonden'` — die view voedt ook de fysieke rol-uitvoer + packer). Nieuwe `geannuleerd`-cascade-effecten = aanhaken op `order_events` (ADR-0006/0015-patroon), géén edit in `markeer_geannuleerd`. **Mig 480 (2026-06-23) — ontbrekende derde tak, zending/pickronde:** geen van de twee bestaande listeners raakte `zendingen`/`zending_regels`/`zending_colli`/`zending_orders` — een order met een actieve pickronde die direct geannuleerd werd, liet de zending als weeskind staan (gevonden tijdens testen; bevestigd als bug door gebruiker: "als een order wordt geannuleerd dan wordt alles geannuleerd... de aangemaakte pickronde/zending wordt verwijderd"). `trg_order_events_zending_release` verwijdert nu, per zending van de geannuleerde order met status **`'Gepland'`/`'Picken'`** (bewust niet verder — `markeer_geannuleerd` blokkeert alleen op `status='Verzonden'`, niet `'Deels verzonden'`, dus een al-fysiek-verzonden deel-zending van een verder geannuleerde order mag nooit geraakt worden, vergelijkbaar met `annuleer_pickronde`'s eigen scope-beperking), de regels/colli van DIE order. Bundel-zending-bewust (mig 222): blijft de zending gekoppeld aan een andere, niet-geannuleerde order, dan blijft de zending zelf bestaan met herberekende `aantal_colli`/`totaal_gewicht_kg`; was de geannuleerde order de enige, dan vervalt de hele zending. Geen "niets-gepickt"-guard (anders dan `annuleer_pickronde`) — annuleren is definitiever dan de "per ongeluk gestart"-correctieknop. Geverifieerd via rolled-back transacties op zowel een solo- als een gefabriceerde bundel-zending (2 orders, 1 geannuleerd → andere orders regels/colli + herberekende aantallen blijven exact intact). Eenmalige backfill in dezelfde migratie ruimde het op dat moment enige bestaande weeskind (ZEND-2026-0197) op.
- **Confectie-planning:** bron is `snijplannen` via view `confectie_planning_forward`. Lane (type_bewerking) wordt afgeleid van `maatwerk_afwerking` via `afwerking_types.type_bewerking`. Afwerkingen `ON`/`ZO` hebben geen lane en verschijnen onder "alleen stickeren". Status-transities lopen via RPC's `start_confectie` en `voltooi_confectie`, niet directe UPDATE.
- **`order_status` 'Wacht op inkoop'/'Wacht op voorraad' — betekenis omgedraaid (mig 470, 2026-06-23):** vóór mig 470 betekende `'Wacht op inkoop'` "er is al een actieve IO-claim, wacht op binnenkomst" en `'Wacht op voorraad'` "geen IO-claim, wel tekort" — het omgekeerde van wat de naam suggereert, en van wat gebruikers verwachten. **Sindsdien (huidig, correct):** `'Wacht op inkoop'` = nog géén inkooporder-claim, er moet één besteld worden; `'Wacht op voorraad'` = inkooporder-claim bestaat al, wacht alleen op levering. Single source of truth: SQL `derive_wacht_status()` (mig 346/352, branches 2/3 omgedraaid) + TS-spiegel `deriveWachtStatus()` (`supabase/functions/_shared/order-lifecycle/derive-status.ts`, golden-fixture-contracttest). De enige andere live plek die hier specifiek op filterde, `trg_io_regel_insert_swap_evaluate()` (mig 297, ADR-0027 — alleen orders zónder IO-claim heralloceren bij een nieuwe IO-regel, anders cascade-swap-risico) is in dezelfde migratie meegedraaid. **Bestaande orders zijn gebackfilld** (één atomaire `CASE`-UPDATE, geen tussentijdse overlap) — de zichtbare status van orders die vóór 23-06-2026 al in een van deze twee statussen stonden is dus ook omgewisseld. **Bewust niet aangeraakt:** de `snijplan_status`-enum heeft toevallig óók een waarde `'Wacht op inkoop'` (mig 437-445, snijplan-niveau IO-koppeling voor maatwerk-stukken) — dat is een volledig los enum-type voor een ander concept en draagt zijn oorspronkelijke (correcte) betekenis nog steeds.
- **Reservering op inkoop (vaste maten, mig 144–152):** orderregels zonder voldoende voorraad alloceren automatisch op openstaande inkooporderregels (status `Besteld`/`Deels ontvangen`, `eenheid='stuks'`) op `verwacht_datum ASC`. Bron-van-waarheid: tabel `order_reserveringen` met rijen voor zowel voorraad- als IO-claims. Centrale RPC: `herallocateer_orderregel(id)` — idempotent, wordt vanuit triggers en handmatig aangeroepen. **Claim-volgorde-prio**: wie eerst claimt, wordt eerst beleverd; geen automatische herallocatie bij urgentere nieuwe orders — **met één gerichte uitzondering: claim-swap (ADR-0027)**. Wanneer een nieuwe urgente orderregel voorraad-tekort heeft, mag de allocator een voorraad-claim van een andere order verplaatsen naar een IO, mits (1) die andere order's `afleverdatum > standaard_afleverdatum_berekend` (= operator heeft bewust later ingevuld) én (2) er een IO bestaat met `verwacht_datum + inkoop_buffer_weken_vast ≤ A.afleverdatum`. Selectie bij meerdere kandidaten: **EDD** (meeste headroom verliest); IO-keuze: **laatst-passend** (bewaart vroege IO's voor toekomstige urgent). Eénstaps, geen cascade. Audit-trail via `order_events`-rij `'claim_geswapt'`. Géén nieuwe kolommen — hergebruikt bestaande `afleverdatum` en `standaard_afleverdatum_berekend` (ADR-0020-snapshot). Bij IO-annulering schuiven claims naar volgende IO of orders gaan in "Wacht op nieuwe inkoop" (orderregel zonder dekking — order-status blijft `Wacht op inkoop` totdat een nieuwe IO de allocator opnieuw triggert via orderregel-bewerking).
- **Maatwerk levertijd-indicator:** op de **orderregel** reserveert maatwerk nog steeds NIET op inkoop — alleen een hint `Eerstvolgende inkoop wk + 2 weken` als er geen rol beschikbaar is. Op **snijplan-niveau** bestaat sinds mig 437-445 wél een echte claim (zie volgende bullet) — twee losse mechanismes; de orderregel-hint is ongewijzigd.
- **Afgeleide snijdatum/rol zichtbaar op order-detail + orderoverzicht, en automatische verzendweek bij materiaal op voorraad (mig 469, 2026-06-23):** de queue-simulatie (`berekenAgenda`) die de Haalbaarheid-pagina al gebruikte (`/snijplanning/haalbaarheid`) leeft nu in gedeelde hook `useSnijHaalbaarheid` (`frontend/src/modules/snijplanning/hooks/use-snij-haalbaarheid.ts`) — order-detail toont per snijplan-stuk de afgeleide snijdatum naast de bestaande rol-badge (`SnijplanStatusBadge`), het orderoverzicht toont een compact label onder de Status-pil voor orders met een open maatwerk-stuk. Géén nieuwe berekening, puur hergebruik (de Haalbaarheid-pagina zelf is een pure refactor op de hook, ongewijzigd gedrag). **Automatische verzendweek:** zodra ALLE snijplan-stukken van een maatwerk-regel "gedekt" zijn — een echte `rol_id` (materiaal op voorraad) ÓF (sinds mig 471) een `verwacht_inkooporder_regel_id` (snijplan-status `'Wacht op inkoop'`, mig 437-445 — de XOR-constraint `snijplannen_rol_of_verwacht_xor` garandeert dat een stuk nooit beide tegelijk heeft) — zet trigger `trg_snijplan_rol_toegewezen_auto_verzendweek` (AFTER INSERT/UPDATE OF `rol_id`, `verwacht_inkooporder_regel_id` op `snijplannen`) **eenmalig** `order_regels.verzendweek`, **mits nog NULL** (nooit een bestaande waarde overschrijven). Bij uitsluitend rol-gedekte stukken: "vandaag + N1 weken" (`app_config.productie_planning.maatwerk_voorraad_levertijd_weken`, default 7) — ongewijzigd mig-469-gedrag. Zijn er ook IO-gedekte stukken: `GREATEST(vandaag + N1, MAX(IO-eta over de IO-gedekte stukken) + N2 weken)` — N2 = bestaande `app_config.order_config.inkoop_buffer_weken_maatwerk` (2, hergebruikt, geen nieuwe config-key). Getest op de live DB (toegevoegd dat een verre IO-ETA de 7-weken-floor inderdaad overstemt, en dat een nabije IO-ETA er onder blijft). **Bewust niet gebouwd:** als een al-gekoppelde IO-regel's ETA later wijzigt, wordt een al-gezette `verzendweek` niet herzien (snapshot, net als de rol-tak — mirrort niet mig 326, die alleen voor de vaste-maat-IO-claim-keten geldt). `verzendweek_bron` (`'handmatig'`/`'automatisch_voorraad'`) onderscheidt een systeemvoorstel van een bewuste keuze. Bewust een **snapshot**, geen live herberekening (anders verschuift de getoonde week elke dag mee, zonder dat er iets veranderd is — onbruikbaar als toezegging richting de klant). `set_regel_verzendweek` (mig 334) labelt voortaan ook `verzendweek_bron`. **Niet te verwarren** met `app_config.order_config.maatwerk_weken` (default 4): die zet de order-brede `afleverdatum` bij order-aanmaak voor élke maatwerk-order ongeacht voorraad — de nieuwe mechaniek is per-regel en specifiek voor het "materiaal is al beschikbaar"-geval, bestaat ernaast. De order-detail `VerzendweekCell`-UI (mig 334) stond alleen open voor niet-maatwerk-regels, nu ook voor maatwerk (live "vandaag + N weken"-voorstel zodra alle stukken al een rol hebben, vóór de trigger zelf gevuurd heeft). Orderbevestiging (`stuur-orderbevestiging`/`_shared/orderbevestiging-pdf.ts`) toont een maatwerk-regel met een eigen `verzendweek` per regel (e-mail: aanvullende sub-regel; PDF: vervangt voor díe regel de order-brede verzendweek-sub-regel die voorheen op elke rij herhaald werd), 4-talig. **`verzendweek_bron` = klantbelofte-barrière:** `order_regels.verzendweek` met `verzendweek_bron='automatisch_voorraad'` is een **berekend voorstel** dat de trigger of de werklijst-UI zet — géén klantbelofte. De waarde wordt pas een belofte zodra een medewerker bewust de orderbevestiging verstuurt (`stuur-orderbevestiging`/ORDRSP). Alleen maatwerk-regels krijgen ooit `bron='automatisch_voorraad'` (beide triggers bewaken `AND is_maatwerk = TRUE`); vaste-maat-regels nooit. **Waarschuwing in de bevestig-flow (nu gebouwd):** `VerzendweekCell` toont een amber "berekend"-label (met `Calculator`-icon) als `bron='automatisch_voorraad'`, zodat de waarde visueel te onderscheiden is van een bewust gezette week. `BevestigOrderDialog` toont een amber waarschuwing als ≥1 maatwerk-regel `bron='automatisch_voorraad'` heeft; de "Verstuur bevestiging"-knop blijft klikbaar (geen hard-block — de bevestiger beslist zelf). Prop `maatwerkMetVoorstelWeek` wordt in `order-detail.tsx` berekend uit de geladen `regels` en via `OrderHeader` doorgegeven. Zie ook mig 526 (handmatige IO-koppeling via werklijst) — de `ontkoppel_snijplan_van_io`-RPC wist een automatisch-bron-week via een clearing arm als de orderregel niet meer volledig gedekt is.
- **Snijplan-tekorten koppelen aan openstaande rol-inkoop (mig 437-445, 2026-06-19):** een snijplan-stuk zonder fysieke rol kan via `auto-plan-groep`'s **tweede pak-pas** alsnog gekoppeld worden aan een openstaande `inkooporder_regel` (eenheid='m', **exacte** kwaliteit_code ÉN kleur_code, FIFO op `verwacht_datum`) — status `'Wacht op inkoop'` (mig 437), `snijplannen.verwacht_inkooporder_regel_id` (mig 438, wederzijds exclusief met `rol_id`). De packer bouwt hiervoor een **in-memory virtuele rol** (`Roll`-object, `sort_priority=3` = altijd ná echte voorraad) — **nooit** een rij in `rollen` (les uit de afgeschafte PH-placeholder-rollen, mig 112/113→182). `RollResult.gebruikte_lengte_cm` wordt 1-op-1 weggeschreven als snapshot op `inkooporder_regels.snijplan_gebruikte_lengte_cm` (RPC's `claim_wacht_op_inkoop`/`release_wacht_op_inkoop_stukken`, release-dan-herberekenen zoals `release_gepland_stukken`; `snijplan_wacht_naar_snijden`-trigger normaliseert `'Wacht'` verder naar `'Gepland'` bij release — bestaand legacy-gedrag, geen aparte status nodig). `snijplanning_tekort_analyse()` (mig 439) sluit deze status uit — de groep verdwijnt uit Tekort; nieuwe RPC `snijplanning_wacht_op_inkoop_analyse()` (mig 440) voedt een eigen tab op de Snijplanning-pagina + een badge op Inkooporders-Regeloverzicht (mig 444). **Bugfix mig 445 (CISC kleur 48, 2026-06-19):** `fetchOpenInkoopRegels` filterde initieel alleen op kwaliteit_code, niet op kleur_code — bij een kwaliteit met tientallen kleuren in dezelfde inkooporder matchte de FIFO-sortering soms de verkeerde kleur. Filter nu ook op `kleur_code` (incl. `.0`-variant, zelfde patroon als `fetchStukken`). Mig 445 fixte daarnaast een losse `RETURNING`-bug in `release_wacht_op_inkoop_stukken` (gaf de net-genulde nieuwe waarde terug i.p.v. de oude, dus `snijplan_gebruikte_lengte_cm` werd nooit teruggezet — nu via een losse pre-update SELECT-CTE). **Scope v1:** geen cross-kwaliteit-matching via uitwisselbare paren (zou de "resterend"-teller tussen groepen kunnen laten conflicteren). pg_net-triggers (mig 441/442, spiegelt mig 100/111) her-plannen bij inkoop-/annuleringswijzigingen maar zijn **inert** tot `app_config.snijplanning.auto_planning.edge_url`/`auth_header` gevuld zijn (bestaande leemte, niet hier geïntroduceerd) — de al-actieve paden (order-/snijplan-aanmaak, "Auto-plan opnieuw draaien"-knop) activeren de nieuwe logica direct.

Plan: [`docs/superpowers/plans/2026-06-18-factuur-concept-fase-uitgestelde-verzending.md`](docs/superpowers/plans/2026-06-18-factuur-concept-fase-uitgestelde-verzending.md). **Pakbon meegestuurd:** [`factuur-verzenden`](supabase/functions/factuur-verzenden/index.ts) hangt per zending die de factuur dekt (per_zending = 1, wekelijks = N, gevonden via `zending_orders` op de gefactureerde orders) een pakbon-PDF aan de mail (debiteur + betaler-kopie), gerenderd door de gedeelde server-side laag [`_shared/pakbon/`](supabase/functions/_shared/pakbon/) (`bouwPakbonDocument` → `genereerPakbonPDF`, pdf-lib — overgenomen uit de onafgemaakte verzendbevestiging-branch, zelfde bron als de geprinte pakbon). **Best-effort:** geen zending / geen colli / render-fout → gelogd + overgeslagen, de factuur-mail gaat altijd door; pakbon best-effort geüpload naar `facturen/{debiteur_nr}/pakbon/{zending_nr}.pdf` voor de e-mailtijdlijn-ref. **Niet te verwarren** met de aparte verzendbevestiging-mail (pakbon naar afleveradres, eigen branch — hier alleen de renderer hergebruikt).

Centrale regel: SQL `effectief_btw_pct(verlegd, pct)` ↔ TS [`_shared/btw.ts`](supabase/functions/_shared/btw.ts) (`effectiefBtwPct`/`isBtwVerlegd`, seam-patroon debiteur-matcher). `genereer_factuur_voor_bundel` (enige live factuur-RPC, mig 240 dropte de rest) schrijft snapshot `facturen.btw_verlegd`; factuur-PDF + orderbevestiging (mail + PDF, 4-talig) tonen dan "BTW verlegd — btw-nr afnemer: X" i.p.v. een BTW-regel. Ontbrekend btw-nummer blokkeert NIET (bewuste keuze) — amber waarschuwing op klant-facturering-tab + script `import/check_verlegd_zonder_btw_nummer.py`. EDI-factuur-mapper checkte de vlag al (ongewijzigd). `debiteuren.btw_percentage` op 0 zetten is NIET meer de route voor EU-klanten — gebruik de verlegd-toggle. **Btw_verlegd is sinds mig 428 ook de daadwerkelijke factuur-aanmaak-RPC `projecteer_concept_factuur`** (+ legacy `genereer_factuur(_voor_week)`) — een regressie in mig 428 liet de verlegd-snapshot/het effectieve tarief daar wegvallen (24 al-verstuurde facturen ten onrechte 21% i.p.v. 0%, € 1.387 te veel berekend, gecorrigeerd 2026-06-20), gefixt in mig 449/453.
- **BTW-regeling per order, afleverland-bewust (mig 454-456, 2026-06-20):** de statische klant-checkbox (vorige bullet) bepaalt het tarief per DEBITEUR, niet per order — geen koppeling met `orders.afl_land`, geen onderscheid EU-ICL vs. export buiten de EU. Nieuwe pure functie `bepaal_btw_regeling(afl_land, debiteur_land, afhalen, verlegd_vlag, btw_nummer, btw_percentage)` (mig 455, TS-spiegel `bepaalBtwRegeling` in `_shared/btw.ts`) combineert het effectieve afleverland (`afl_land`, fallback `debiteuren.land`) met de checkbox tot 4 regelingen: `nl_binnenland` (incl. **leeg land bij order én debiteur** — bewust géén blokkade, 62% van de actieve debiteuren heeft een leeg land-veld, legacy NL-klanten), `eu_b2b_icl` (0%, advisory bij ontbrekend btw-nummer — mig 164-besluit niet heropend), `eu_b2b_binnenland_afwijking` (EU-land maar checkbox staat uit — mismatch, **hard-block**), `export_buiten_eu` (buiten de EU, 0% mits exportbewijs — **hard-block**, geen exportbewijs-tracking dus altijd menselijke bevestiging). `is_eu_land(iso2)` (mig 454, hardcoded 27-lidstatenlijst, CH/NO/GB bewust non-EU) vereiste eerst uitbreiding van `normaliseer_land` met 13 ontbrekende EU-landnamen (Portugal, Slowakije, Hongarije, Griekenland, Slovenië, Estland, Letland, Litouwen, Bulgarije, Roemenië, Kroatië, Cyprus, Finland — zat al in mig 164's eenmalige backfill-lijst, nooit structureel in `normaliseer_land`). De factuur-aanmaak-RPC's (`projecteer_concept_factuur`/`genereer_factuur(_voor_week)`, mig 456: `facturen.btw_controle_nodig_sinds` + `btw_regeling`) zetten de gate-kolommen **altijd**, ook bij een hard-block-regeling — de factuur wordt dus **altijd aangemaakt** (zichtbaar als Concept met de banner). **Correctie zelfde dag (vóór merge):** de eerste versie liet de RPC's zelf RAISE EXCEPTION doen vóór de INSERT, waardoor er bij een blokkade HELEMAAL GEEN factuur ontstond — alleen `factuur_queue.last_error` (een tabel zonder UI), dus de banner was onbereikbaar. Fix: **de blokkade zit nu in `factuur-verzenden/index.ts`**, ná het aanmaken van de factuur en vóór het versturen van mail/EDI (checkt `HARD_BLOCK_REGELINGEN` uit `_shared/btw.ts`) — niet op `_valideer_intake_gates`/pick & ship, het risico is een verkeerd BTW-bedrag, niet het fysiek verzenden; een order mag gewoon gepickt/verzonden worden, de factuur blijft op Concept tot bevestigd via `markeer_btw_regeling_geaccepteerd` (UI: `BtwControleNodigBanner` op factuur-detail + banner/filter op het facturen-overzicht). Aanleiding: 6 actieve niet-EU-debiteuren (VS/Australië/Suriname/Ukraine/VK) stonden op 21% i.p.v. 0%-export. **Scope bewust niet meegenomen:** VIES, OSS/particulier (bevestigd: vrijwel uitsluitend B2B), exportbewijs-documentatie, ICP-automatisering, binnenlandse-verlegging (niet relevant voor tapijt-groothandel).
- **Universele bevestig-knop (plan 2026-06-11):** "Bevestig order" dispatcht op `bron_systeem` — EDI-order → ORDRSP via `useBevestigEdiOrder` (gate `orderbev_uit`; `edi_stil` = alleen `edi_bevestigd_op` zetten), anders e-mail via `stuur-orderbevestiging`. EDI-orders krijgen nooit een e-mail-orderbevestiging. Eén bevestigd-predicaat: `isOrderBevestigd` in `frontend/src/lib/orders/bevestiging-kanaal.ts`.
- **Factuurdocument als deep module (ADR-0036, 2026-06-14):** een factuur wordt naar buiten gerenderd via drie paden — factuur-PDF, EDI-INVOIC automatisch ([`factuur-verzenden`](supabase/functions/factuur-verzenden/index.ts)) én handmatig ([`bouw-factuur-edi`](supabase/functions/bouw-factuur-edi/index.ts)). Die leidden de regel-inhoud elk **onafhankelijk** af → dezelfde factuur gaf een ánder INVOIC afhankelijk van auto vs. handmatig (handmatig was kaler: géén karpi_code/klant_artikel/gewicht), en de PDF toonde een derde artikeltekst-variant. **Eén canoniek [`FactuurDocument`](supabase/functions/_shared/facturatie/factuur-document.ts)** (`fetchFactuurDocument` → header + regels mét [`Artikelpresentatie`](supabase/functions/_shared/facturatie/artikel-presentatie.ts) + effectief BTW-tarief via de `btw.ts`-seam) voedt nu drie **dunne renderers**: `naarInvoiceInput` (vervangt zowel `buildEdiFactuurInput` als het verwijderde `factuur-mapper.ts` → beide EDI-paden byte-identiek, golden-gepind) en `naarFactuurPdfInput` (factuur-PDF + `factuur-verzenden`-PDF; toont nu de opgeloste artikeltekst incl. karpi_code, **zelfde tekst als de EDI-articleDescription**). De **orderbevestiging** ([`stuur-orderbevestiging`](supabase/functions/stuur-orderbevestiging/index.ts)) deelt alléén de resolver (`resolveKarpiCode`; BTW deelde het al via `btw.ts`) maar houdt een eigen document — ander lifecycle-moment (order-tijd, `order_regels`, 4-talige vertaling). **Nieuw factuur-renderpad = dunne renderer op het document, géén eigen fetch/resolve.** Vangnet: golden `KarpiInvoiceInput` byte-voor-byte + pure-functie-tests (`_shared/facturatie/`). **Deploy-fan-out:** `factuur-verzenden`/`bouw-factuur-edi`/`factuur-pdf`/`stuur-orderbevestiging` delen `_shared/facturatie/` → alle vier herdeployen.
- **Vervoerder-capability-seam (ADR-0034, 2026-06-13):** de *keuze*-as (welke vervoerder een zending krijgt) is data-driven via `vervoerder_selectie_regels` (ADR-0008/0030); de *capability*-as (wat een vervoerder EIST en KAN — landbereik, preflight-velden, default-afmetingen, protocol, batch-limiet) staat nu op **één plek**: de pure registry [`_shared/vervoerders/capabilities.ts`](supabase/functions/_shared/vervoerders/capabilities.ts) (`VERZEND_CAPABILITIES` / `capabilityVoor(code)` → `VerzendCapability`). Consumers lezen de descriptor i.p.v. eigen `if code === `-takken: `valideerVoorVervoerder` ([`_shared/vervoerder-eisen.ts`](supabase/functions/_shared/vervoerder-eisen.ts), adres/telefoon/land), `valideerColli` ([`_shared/vervoerders/colli.ts`](supabase/functions/_shared/vervoerders/colli.ts), per-colli velden + 0-colli-guard — de carriers leveren alleen de meldingstekst, zo blijven de subtiele verschillen exact: Verhoek eist breedte, Rhenus niet maar wél ≥1 colli/incident 0455395), de HST `DEFAULT_*`-afmetingen in `payload-builder.ts`, en de drie `MAX_PER_RUN`. Pure registry (géén DB) zodat de frontend hem via de shim deelt (ADR-0033); de `vervoerders`-tabel blijft de administratieve bron (`actief`/`display_naam`/FK). **Vierde vervoerder = één registry-rij + één format-adapter** (+ routering als data), géén sweep. **Buiten scope:** de format-builders zelf (echte protocolverschillen, by-design), de orchestrator-loop-skeletten (aparte "process-as"-sibling-seam → ADR-0035, zie hieronder), en `vervoerders.type`-correctie + SQL↔TS-contracttest (slice 4, optioneel). Plan: [`docs/superpowers/plans/2026-06-13-vervoerder-capability-seam.md`](docs/superpowers/plans/2026-06-13-vervoerder-capability-seam.md).
- **Verzend-orchestrator-skeleton (ADR-0035, process-as, 2026-06-14):** de drie verzend-edge-functions (`hst-send`/`verhoek-send`/`rhenus-send`) deelden een vrijwel identiek per-rij-skelet (fetch zending/order/bedrijf → colli → 0-colli-guard → preflight → bestandsnaam → render → transport → audit → markeer); alleen render + transport zijn carrier-specifiek. Dat skelet leeft nu één keer in [`_shared/verzend-orchestrator.ts`](supabase/functions/_shared/verzend-orchestrator.ts) (`verwerkVerzendRij(adapter, …)`); elke carrier levert een `VerzendAdapter` (in zijn `verwerk-row.ts`) met de carrier-specifieke stukken (kanaal, select-kolommen, builder, transport, markeer-RPC's, storage). De `verwerkRow`-export per carrier is nu een dunne wrapper naar de skeleton. **Vangnet = karakterisatie-tests** (`*-send/verwerk-row.test.ts`, 15 stuks) met een herbruikbare [`_shared/__tests__/fake-supabase.ts`](supabase/functions/_shared/__tests__/fake-supabase.ts)-recorder die de side-effect-sequence (rpc/update/storage) vastlegt; die bleven over alle slices ongewijzigd groen → gedragsneutraal. **Bewust buiten de skeleton:** de claim-loop + reaper + secret/dry-run-resolutie blijven in elke `index.ts` (carrier-specifieke env-resolutie). Plan: [`docs/superpowers/plans/2026-06-14-verzend-orchestrator-skeleton-seam.md`](docs/superpowers/plans/2026-06-14-verzend-orchestrator-skeleton-seam.md).

## Conventies
- Taal in code: Engels (variabelen, functies). Taal in UI: Nederlands.
- Bedragen: € 1.234,56 (Nederlands formaat)
- Datums: DD-MM-YYYY in UI, ISO in database
- Status badges: kleurgecodeerd per enum waarde
- Queries: per module in `frontend/src/lib/supabase/queries/`
- **Bestanden klein houden:** splits logisch op als een bestand >200-300 regels groeit
- Componenten: 1 concern per bestand, extracteer herbruikbare delen
- **Gedeelde TS-logica edge ↔ frontend (ADR-0033):** pure modules leven éénmalig in `supabase/functions/_shared/`; de frontend importeert/re-exporteert ze cross-root (zie de shims `@/lib/utils/iso-week`, `@/lib/utils/snijplan-status`, `@/lib/email-recipients`, `@/lib/orders/vervoerder-eisen`, en `@/lib/utils/land-vlag` dat `landNaarIso2Strikt` uit `_shared/adres-split.ts` re-exporteert). Nieuwe gedeelde logica wordt **nooit gekopieerd**; kan het niet puur (Deno-API's/https-imports/DB), dan aparte modules per runtime + golden-file-contracttest (patroon `derive-status.golden.json`).
- **Landnaam→ISO-2-normalisatie (single source, 2026-06-13):** vrije landnamen ('Nederland', 'BELGIË', 'Österreich') → ISO-2 loopt via **één** seam `_shared/adres-split.ts`: `normalizeCountry` (lenient — onbekend land komt uppercased/diakriet-vrij terug, voor vrachtbrief/EDI/factuur) en `landNaarIso2Strikt` (strikt — onbekend → null, voor de frontend-vlag). Beide spiegelen de SQL-bron `normaliseer_land` (mig 214) één-op-één. **SQL↔TS-contract (mig 389):** golden fixtures `frontend/src/lib/orders/__tests__/golden/normaliseer-land.golden.json` getoetst door `normaliseer-land.contract.test.ts` (Vitest) + `assert_normaliseer_land_contract()`. Wie `normaliseer_land` of de seam wijzigt: golden bijwerken + nieuwe `*_normaliseer_land_contract*.sql`. De oude lokale kopieën in `factuur-verzenden`/`factuur-mapper`/`factuur-pdf` (waarvan één AT/CH/ES/PL/GB verkeerd mapte via `slice(0,2)`) bestaan niet meer.

## Werkwijze met Claude Code (team-tips van Boris Cherny)
Deze werkafspraken verhogen kwaliteit en snelheid. Geen dogma's — experimenteer en behoud wat werkt.

### Verplichte poorten (geen narratief — altijd toepassen)
- **Impact-preflight vóór elke inhoudelijke wijziging:** bepaal welke tabellen/RPC's je raakt, grep de codebase op álle lezers/schrijvers daarvan, en meld welke modules geraakt worden — óók ongenoemde (inkoop, maatwerk, snijplanning, facturatie). De ADR-narratieven hieronder zijn retrospectief; deze stap dwingt dezelfde analyse vooraf af.
- **Runtime-bewijs vóór "klaar":** een wijziging met een DB-schrijfactie is niet klaar tot een test of directe query bewijst dat de rij ECHT veranderde, en elke geraakte UI-knop getraceerd is knop→handler→RPC→tabel. "De code/trigger staat er" is geen bewijs. Dit naast de bestaande golden/contract-tests, niet in plaats daarvan.

### Planning & uitvoering
- **Verticaal implementeren als standaard:** bouw features waar mogelijk in dunne verticale slices — één samenhangend stukje functionaliteit end-to-end door alle lagen (DB/migratie → RPC/edge function → query → UI), zodat het meteen werkt en testbaar is. NIET horizontaal (eerst alle DB, dan alle backend, dan alle frontend). Elke slice levert werkende, demonstreerbare waarde op; pas als verticaal echt niet kan, val terug op een horizontale aanpak.
- **Plan-mode eerst bij niet-triviale taken:** maak eerst een solide plan, laat daarna implementeren. Bij complexe features: één Claude schrijft plan, een tweede reviewt als "staff engineer". Loopt het mis → terug naar plan-mode, niet doorduwen.
- **Detailleer specs:** verwijder ambiguïteit vóór overdracht. Specificiteit = autonomie.
- **Daag output uit:** vraag Claude om keuzes te rechtvaardigen ("prove this works", vergelijk main vs branch). Middelmatige fix? Vraag om verse herschrijving ("scrap it, implement the elegant solution").

### Git-workflow (branch-strategie)
- **Grotere wijzigingen krijgen automatisch een eigen branch.** Begin substantieel werk (nieuwe feature, refactor over meerdere bestanden, migratie, alles met >~1 logische stap of meerdere bestanden) zónder te vragen op een eigen branch — bv. `feat/<korte-naam>`, `refactor/<korte-naam>`, `fix/<korte-naam>`. Commit het werk daar, en **merge pas naar `main` wanneer ik dat expliciet zeg** ("merge maar", "naar main"). Niet automatisch mergen.
- **Kleine, triviale wijzigingen** (één-regel-fix, typo, docs-tweak) mogen direct op `main` — gebruik je oordeel; bij twijfel: aparte branch.
- **Reden:** er draaien vaak meerdere Claude-sessies tegelijk in deze working tree. Direct op `main` werken laat sessies elkaars ongecommitte werk overschrijven/oppakken (zie incident 2026-06-07: gedeelde `changelog.md`/`architectuur.md` raakten verstrengeld). Een eigen branch per substantiële taak isoleert het werk tot ik bewust merge.
- **Merge-moment:** bij "merge naar main" → `git checkout main && git pull --ff-only && git merge <branch>` (of fast-forward), daarna push. Los merge-conflicten op de branch op, niet op `main`.
- **Auto-opruimen na merge:** zodra een branch naar `main` is gemerged én die merge live op de remote staat (gepusht), ruim je het lokale werk automatisch op zonder te vragen: verwijder de bijbehorende git-worktree (`git worktree remove <pad>`, gebruik `--force` als er alleen rommel als `.pyc`/test-output instaat) **plus** de gemergde branch (`git branch -d <branch>`), en `git worktree prune`. Verifieer "gemerged" met `git merge-base --is-ancestor <branch> origin/main` vóór verwijderen. **Nooit** opruimen: de actieve working dir, niet-gemergde branches/worktrees, of `locked` worktrees (mogelijk actieve sessies). Op Windows kan `git worktree remove` falen met "Invalid argument" terwijl de admin wél verdwijnt — wis dan de achtergebleven map met `Remove-Item -Recurse -Force` en draai `git worktree prune`.
- Géén PR's nodig (tenzij ik erom vraag) — branch + merge-op-commando volstaat.
