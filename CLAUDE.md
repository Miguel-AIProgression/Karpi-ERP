ERP-portaal voor Karpi: beheer van klanten (debiteuren), producten (tapijtrollen), orders, en operationele processen. Gebouwd op Supabase + React/TypeScript.

- **Frontend:** React 18+, TypeScript, Vite, TailwindCSS, shadcn/ui, TanStack Query

## Bedrijfsregels
- **Orders verwijderen:** mag alleen als er geen snijplannen zijn met status 'Snijden', 'Gesneden' of later. Snijplannen met vroege statussen ('Wacht', 'Gepland') moeten eerst handmatig verwijderd worden v├│├│r de order. Gooit anders FK-fout op `snijplannen_order_regel_id_fkey`.
- **Orders annuleren cascadeert naar Snijplanning (ADR-0023, mig 290):** `markeer_geannuleerd` (mig 218) schrijft een `geannuleerd`-event in `order_events`; daarop reageren twee ontkoppelde listeners ŌĆö `trg_order_events_reservering_release` (mig 255, releaset voorraad+IO-claims) ├®n `trg_order_events_snijplan_release` (mig 290). De tweede zet **alle** snijplannen van de order op `Geannuleerd` (ongeacht voortgang ŌĆö bewuste werkvloer-keuze, anders dan bij _verwijderen_) en geeft geraakte rollen die hun laatste actieve (`Gepland/Snijden/Gesneden`) snijplan verliezen vrij naar `beschikbaar`/`reststuk` (`snijden_gestart_op=NULL`), met `NOT EXISTS`-guard zodat een rol die nog een andere order bedient onaangeroerd blijft. Defense-in-depth: `snijplanning_overzicht` filtert sinds mig 290 `WHERE o.status <> 'Geannuleerd'` (bewust NIET ook `'Verzonden'` ŌĆö die view voedt ook de fysieke rol-uitvoer + packer). Nieuwe `geannuleerd`-cascade-effecten = aanhaken op `order_events` (ADR-0006/0015-patroon), g├®├®n edit in `markeer_geannuleerd`.
- **Confectie-planning:** bron is `snijplannen` via view `confectie_planning_forward`. Lane (type_bewerking) wordt afgeleid van `maatwerk_afwerking` via `afwerking_types.type_bewerking`. Afwerkingen `ON`/`ZO` hebben geen lane en verschijnen onder "alleen stickeren". Status-transities lopen via RPC's `start_confectie` en `voltooi_confectie`, niet directe UPDATE.
- **Reservering op inkoop (vaste maten, mig 144ŌĆō152):** orderregels zonder voldoende voorraad alloceren automatisch op openstaande inkooporderregels (status `Besteld`/`Deels ontvangen`, `eenheid='stuks'`) op `verwacht_datum ASC`. Bron-van-waarheid: tabel `order_reserveringen` met rijen voor zowel voorraad- als IO-claims. Centrale RPC: `herallocateer_orderregel(id)` ŌĆö idempotent, wordt vanuit triggers en handmatig aangeroepen. **Claim-volgorde-prio**: wie eerst claimt, wordt eerst beleverd; geen automatische herallocatie bij urgentere nieuwe orders ŌĆö **met ├®├®n gerichte uitzondering: claim-swap (ADR-0027)**. Wanneer een nieuwe urgente orderregel voorraad-tekort heeft, mag de allocator een voorraad-claim van een andere order verplaatsen naar een IO, mits (1) die andere order's `afleverdatum > standaard_afleverdatum_berekend` (= operator heeft bewust later ingevuld) ├®n (2) er een IO bestaat met `verwacht_datum + inkoop_buffer_weken_vast Ōēż A.afleverdatum`. Selectie bij meerdere kandidaten: **EDD** (meeste headroom verliest); IO-keuze: **laatst-passend** (bewaart vroege IO's voor toekomstige urgent). E├®nstaps, geen cascade. Audit-trail via `order_events`-rij `'claim_geswapt'`. G├®├®n nieuwe kolommen ŌĆö hergebruikt bestaande `afleverdatum` en `standaard_afleverdatum_berekend` (ADR-0020-snapshot). Bij IO-annulering schuiven claims naar volgende IO of orders gaan in "Wacht op nieuwe inkoop" (orderregel zonder dekking ŌĆö order-status blijft `Wacht op inkoop` totdat een nieuwe IO de allocator opnieuw triggert via orderregel-bewerking).
- **Maatwerk levertijd-indicator:** maatwerk reserveert NIET op inkoop in V1. Op de orderregel verschijnt alleen een hint `Eerstvolgende inkoop wk + 2 weken` als er geen rol beschikbaar is. Echte claim op rol-IO (`eenheid='m'`) staat op de V2-backlog.
- **FIFO-magazijnleeftijd in de snijplanner (ADR-0021, mig 280-286):** kleurverschil tussen rollen van dezelfde kwaliteit+kleur ontstaat puur door fysieke veroudering. Bron-van-waarheid is `rollen.in_magazijn_sinds` (NIET `reststuk_datum` ŌĆö dat reset bij snijden en wordt ook voor volle IO-rollen op `NOW()` gezet). Reststukken/aangebroken rollen **erven** `in_magazijn_sinds` van de moederrol. **Modus-schakelaar `app_config.snijplanning.modus` (mig 285, default `'simpel'`):** in `simpel` (huidig live) pakt de packer strikt de oudst-binnengekomen rol eerst ŌĆö g├®├®n kost-afweging/badge/carve-out (`fifoMetrics` leeg); de rollen-pagina toont per (kw,kl) binnenkomstdatum + "1e binnen" (mig 286, `voorraadposities` sorteert oudste-eerst). In `geavanceerd` weegt de packer (`_shared/guillotine-packing.ts`) leeftijd als kost: `afval ŌłÆ ╬▒┬Ęmax(0, leeftijd ŌłÆ drempel)`, drempel 90 dgn, absolute voorrang Ōēź180 dgn, tunebaar via `app_config.snijplanning`; **C1** geen verdringing van rollen in een goedgekeurd voorstel; **C2** geen stuk met afleverdatum laten vallen dat efficiency w├®l plaatste; short-circuit bij verse voorraad; **rode** badge ŌåÆ in `auto-plan-groep` NIET auto-goedgekeurd. Zonder `PackOptions.fifo` is het packer-gedrag exact als voorheen. Omschakelen naar `geavanceerd` zodra de interne rol-data op orde is.
- **`lever_modus`:** order-niveau keuze "deelleveringen" / "in_een_keer". Default uit `debiteuren.deelleveringen_toegestaan`, gevuld via `LeverModusDialog` bij opslaan als Ōēź1 regel tekort heeft. Bepaalt levertijd-berekening (eerste IO-week resp. max IO-week) en aantal zendingen. NULL voor orders zonder tekort.
- **`lever_type` (ADR 0014 / mig 244):** order-niveau intentie `'week'` (B2B-default, ~90%) of `'datum'` (B2C). Dag-orders verschijnen pas 1 werkdag v├│├│r afleverdatum in Pick & Ship (helper `werkdagMinN` in de werkagenda-kernel [`_shared/werkagenda.ts`](supabase/functions/_shared/werkagenda.ts); frontend importeert die direct via `bereken-agenda.ts`; horizon + snij-deadline tellen met de centrale werkagenda-config (`app_config 'werkagenda'`, mig 384 ŌĆö feestdagen/werkdagen)) en krijgen voorrang in [`check-levertijd`](supabase/functions/check-levertijd/index.ts) (kritieke deadline = afleverdatum ŌłÆ `app_config.productie_planning.dag_order_snij_buffer_werkdagen`, default 2 werkdagen). Default per klant in `debiteuren.default_lever_type`; per order overschrijfbaar via segmented toggle in [`LeverDatumField`](frontend/src/components/orders/order-form.tsx). Bundel-sleutel (mig 228-230), wekelijkse factuur-cron (mig 231-232) en IO-sync (mig 153) ongewijzigd ŌĆö dag- en week-orders bundelen normaal als de 4D-sleutel matcht. Visuele badges (terracotta ­¤ōģ) op order-detail, Pick & Ship en orders-overzicht onderscheiden dag-orders.
- **`vrije_voorraad`-formule (mig 149):** `voorraad ŌłÆ gereserveerd ŌłÆ backorder` (geen `+ besteld_inkoop` meer). `gereserveerd` is voortaan SUM van actieve `bron='voorraad'`-claims uit `order_reserveringen`, niet meer SUM van `te_leveren` op order_regels. Toekomstige inkoop blijft zichtbaar via `besteld_inkoop` en de claims, maar telt niet meer in "vandaag-leverbaar".
- **Antislip doos-stuks koppeling (mig 408):** antislip-producten worden per doos ingekocht maar aan sommige klanten per doos en aan anderen per stuk aangeboden. Architectuur: **stuks-artikel = bron-van-waarheid** voor voorraad; **doos-artikel = ordering vehicle**. Koppeling via `producten.stuks_artikelnr` + `producten.stuks_per_doos`. Allocator (`herallocateer_orderregel`): als `stuks_artikelnr IS NOT NULL` ŌåÆ vertaal v├│├│r allocatie naar `(stuks_artikelnr, te_leveren ├Ś stuks_per_doos)`. Trigger `trg_sync_doos_vrije_voorraad` houdt `doos.vrije_voorraad = floor(stuks.vrije_voorraad / stuks_per_doos)` automatisch in sync ŌåÆ bestaande UI/order-form werkt zonder aanpassingen. **Inkoop-IOs altijd op het stuks-artikel aanmaken (in stuks), nooit op het doos-artikel** ŌĆö anders matcht de IO-claim-lookup in stap 2 van de allocator niet. Huidige koppelingen: 900000005 (80├Ś150, 20st/doos) Ōåö 900000020; 900000006 (130├Ś190, 15st) Ōåö 900000021; 900000000 (160├Ś230, 12st) Ōåö 900000022; 900000001 (190├Ś290, 8st) Ōåö 900000023; 900000009 (240├Ś340, 5st) Ōåö 900000024. Open: 900000015 (300├Ś400, 4st/doos) wacht op stuks-artikel 900000025; 900000018 (60├Ś110) stuks-only zonder doos. Prijzen stuks = doos_prijs / stuks_per_doos ├Ś 1.25 (25%-toeslag losse stuks).
- **Admin-pseudo-orderregel (ADR-0018, mig 272-273):** orderregels waarvan het artikel administratief is (geen fysieke leverbaarheid) worden uniform geskipt in allocator, status-bepaling, levertijd-view, pickbaarheid en dekking-preview. Bron-van-waarheid: `producten.is_pseudo BOOLEAN` (mig 272). Predikaten: SQL `is_admin_pseudo(artikelnr)` (mig 272) en TS `isAdminPseudo(regel)` ([`@/lib/orders/admin-pseudo`](frontend/src/lib/orders/admin-pseudo.ts) ŌĆö accepteert zowel form-data met `is_pseudo` top-level als query-resultaten met `producten ( is_pseudo )`-join). Nieuwe admin-pseudo toevoegen = pure `UPDATE producten SET is_pseudo=TRUE` ŌĆö geen code-edit. **Niet te verwarren** met `SHIPPING_PRODUCT_ID='VERZEND'`: die constant bedient toe-voegen van een verzendregel (`applyShippingLogic`), niet skip-detectie. De oorspronkelijke driedubbele-fix (mig 263ŌåÆ266ŌåÆ269) en de claim-keten-recursie-bug (mig 263ŌåÆ267) komen categorisch niet meer voor ŌĆö er is geen string-lijst om uit te breiden.
- **Afleverdatum sync met IO-claims (mig 153):** `herwaardeer_order_status` synct na elke alloc-cyclus de `orders.afleverdatum` naar de laatste IO-claim-leverdatum (`MAX(verwacht_datum) + inkoop_buffer_weken_vast ├Ś 7` dagen) als die later valt dan de huidige afleverdatum. Schuift alleen v├│├│ruit, nooit terug ŌĆö de afleverdatum reflecteert dus altijd minimaal de werkelijke claim-belofte aan de klant. Eindstatussen (Verzonden / Geannuleerd / Klaar voor verzending) blijven ongewijzigd.
- **Uitwisselbaar = handmatige claims (mig 154):** orderregel kan multi-source gedekt worden ŌĆö voorraad eigen artikel + voorraad uitwisselbare(n) (omgestickerd) + IO eigen artikel ŌĆö zonder de regel zelf te splitsen. Gebruiker kiest in [`UitwisselbaarTekortHint`](frontend/src/components/orders/uitwisselbaar-tekort-hint.tsx) per uitwisselbaar product een **aantal**; bij submit roept `order-form` `set_uitwisselbaar_claims(regel_id, [{artikelnr, aantal}])` aan die `is_handmatig=true` claims aanmaakt. Allocator (`herallocateer_orderregel`) **respecteert handmatige claims** (releaset alleen niet-handmatige) en vult resterend aan met voorraad eigen + IO eigen. **Voorraad uitwisselbaar product wordt ook daadwerkelijk gereserveerd** ŌĆö `herbereken_product_reservering` telt op `fysiek_artikelnr`. **Factuur en order-regel weergave blijven 1├Ś origineel artikel**; de omstickering is intern. Op order-detail wordt de claim-uitsplitsing per stuks-regel zichtbaar als geneste sub-rijen (eigen voorraad ŌåÆ omsticker ŌåÆ IO ŌåÆ wacht op nieuwe inkoop), met locatie + omschrijving van het uitwisselbare bron-product bij omsticker-rijen ŌĆö gericht op de verzamelaar in het magazijn. Bron: [`fetchClaimsVoorOrder`](frontend/src/lib/supabase/queries/reserveringen.ts) + hook `useClaimsVoorOrder`.
- **Inkoopgroepen (mig 189):** 10 INKC-organisaties (BEGROS, DECOR UNION, FACHHANDELSRING, INTERRING, VME, VME (TH), TINTTO, INHOUSE, HOUSE OF DUTCHZ, MUSTERRING) staan in tabel `inkoopgroepen`. Een debiteur hoort aan **maximaal 1** inkoopgroep via FK `debiteuren.inkoopgroep_code`. `orders.inkooporganisatie` blijft TEXT-snapshot ŌĆö orders bewegen niet mee bij wijziging op debiteur. Centraal beheer onder `/inkoopgroepen`; klant-detail toont de groep als link.

## Werkwijze met Claude Code (team-tips van Boris Cherny)
Deze werkafspraken verhogen kwaliteit en snelheid. Geen dogma's ŌĆö experimenteer en behoud wat werkt.

### Planning & uitvoering
- **Verticaal implementeren als standaard:** bouw features waar mogelijk in dunne verticale slices ŌĆö ├®├®n samenhangend stukje functionaliteit end-to-end door alle lagen (DB/migratie ŌåÆ RPC/edge function ŌåÆ query ŌåÆ UI), zodat het meteen werkt en testbaar is. NIET horizontaal (eerst alle DB, dan alle backend, dan alle frontend). Elke slice levert werkende, demonstreerbare waarde op; pas als verticaal echt niet kan, val terug op een horizontale aanpak.
- **Plan-mode eerst bij niet-triviale taken:** maak eerst een solide plan, laat daarna implementeren. Bij complexe features: ├®├®n Claude schrijft plan, een tweede reviewt als "staff engineer". Loopt het mis ŌåÆ terug naar plan-mode, niet doorduwen.
- **Detailleer specs:** verwijder ambigu├»teit v├│├│r overdracht. Specificiteit = autonomie.
- **Daag output uit:** vraag Claude om keuzes te rechtvaardigen ("prove this works", vergelijk main vs branch). Middelmatige fix? Vraag om verse herschrijving ("scrap it, implement the elegant solution").

### Git-workflow (branch-strategie)
- **Grotere wijzigingen krijgen automatisch een eigen branch.** Begin substantieel werk (nieuwe feature, refactor over meerdere bestanden, migratie, alles met >~1 logische stap of meerdere bestanden) z├│nder te vragen op een eigen branch ŌĆö bv. `feat/<korte-naam>`, `refactor/<korte-naam>`, `fix/<korte-naam>`. Commit het werk daar, en **merge pas naar `main` wanneer ik dat expliciet zeg** ("merge maar", "naar main"). Niet automatisch mergen.
- **Kleine, triviale wijzigingen** (├®├®n-regel-fix, typo, docs-tweak) mogen direct op `main` ŌĆö gebruik je oordeel; bij twijfel: aparte branch.
- **Reden:** er draaien vaak meerdere Claude-sessies tegelijk in deze working tree. Direct op `main` werken laat sessies elkaars ongecommitte werk overschrijven/oppakken (zie incident 2026-06-07: gedeelde `changelog.md`/`architectuur.md` raakten verstrengeld). Een eigen branch per substanti├½le taak isoleert het werk tot ik bewust merge.
- **Merge-moment:** bij "merge naar main" ŌåÆ `git checkout main && git pull --ff-only && git merge <branch>` (of fast-forward), daarna push. Los merge-conflicten op de branch op, niet op `main`.
- **Auto-opruimen na merge:** zodra een branch naar `main` is gemerged ├®n die merge live op de remote staat (gepusht), ruim je het lokale werk automatisch op zonder te vragen: verwijder de bijbehorende git-worktree (`git worktree remove <pad>`, gebruik `--force` als er alleen rommel als `.pyc`/test-output instaat) **plus** de gemergde branch (`git branch -d <branch>`), en `git worktree prune`. Verifieer "gemerged" met `git merge-base --is-ancestor <branch> origin/main` v├│├│r verwijderen. **Nooit** opruimen: de actieve working dir, niet-gemergde branches/worktrees, of `locked` worktrees (mogelijk actieve sessies). Op Windows kan `git worktree remove` falen met "Invalid argument" terwijl de admin w├®l verdwijnt ŌĆö wis dan de achtergebleven map met `Remove-Item -Recurse -Force` en draai `git worktree prune`.
- G├®├®n PR's nodig (tenzij ik erom vraag) ŌĆö branch + merge-op-commando volstaat.
